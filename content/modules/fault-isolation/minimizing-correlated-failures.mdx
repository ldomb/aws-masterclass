# Minimizing Correlated Failures in Distributed Systems

## Introduction: The Promise and Peril of Redundancy

Eighteen years ago, a small startup built a three-tier web application—Java frontend, C++ backend, relational database—all running on a single server. When that server failed, the entire business went offline. Engineers worked urgently to restore service or move the application to a different server. To minimize downtime, an entire ecosystem of expensive servers with redundant, hot-swappable components thrived. Some servers cost as much as the entire company's valuation.

This was the norm. Single points of failure were accepted as unavoidable, and the solution was making individual components as reliable as possible through expensive, specialized hardware. A few years later, Amazon was adopting a radically different approach: service-oriented architectures running on distributed systems built from inexpensive, commodity servers.

The mathematics of redundancy appeared compelling. If a single server has a 0.01% chance of failing on any given day, running on two servers should reduce the probability of complete failure to 0.000001%—transforming a somewhat unlikely event (1 in 10,000 days) into an incredibly unlikely one (1 in 100 million days). By distributing workloads across multiple servers behind load balancers, services could continue operating even when individual components failed.

However, this calculation relies on a critical assumption: **failures must be uncorrelated and independent**. In the real world, this assumption often doesn't hold. A single underlying cause—a shared power system, a common software bug, synchronized behavior patterns—can trigger multiple failures simultaneously, completely undermining the availability gains we expect from redundancy.

Understanding and minimizing correlated failure risks is essential for building truly resilient distributed systems. This lesson explores the various sources of correlation, from obvious infrastructure dependencies to subtle behavioral patterns, and presents practical techniques for reducing these risks while maintaining the efficiency benefits of shared infrastructure.

## The Nature of Correlated Failures

A **correlated failure** occurs when multiple supposedly independent components fail due to the same underlying cause. These failures can range from obvious to subtle, from easy to predict to nearly impossible to anticipate.

### A Household Analogy

Consider a real-world example: A homeowner replaced all kitchen cabinets using identical plastic mounting brackets from the same production batch. The cabinets worked flawlessly for over a year. Then one day, a bracket in the first cabinet failed. A week later, brackets in the second cabinet broke. Then the third, then the fourth. Within three months, every single cabinet's mounting hardware had failed and needed replacement.

What caused this correlated failure? Possible root causes included:
- **Manufacturing defect**: The entire batch had a latent flaw in the plastic formulation
- **Design error**: The brackets were undersized for the actual load they needed to bear
- **Environmental factor**: Moisture levels in the house degraded the plastic over time
- **Usage pattern**: How the family used the cabinets created consistent stress on the brackets

Whatever the specific cause, an unforeseen factor created conditions where supposedly independent components failed together rather than at independent times.

The key insight: The homeowner expected each cabinet to have an independent probability of bracket failure. Instead, all cabinets shared a common cause that made failures correlated. The mathematics of independent probabilities (where failures are rare) was replaced by the reality of correlated probabilities (where all failures happen in a concentrated period).

For cabinets, this was an inconvenience. For distributed systems serving millions of users, correlated failures can be catastrophic.

### How Correlation Undermines Redundancy

Let's examine the mathematics more carefully with a concrete example:

**Assumption: Independent failures**
- Server A: 0.01% daily failure probability
- Server B: 0.01% daily failure probability
- Probability both fail on same day: 0.01% × 0.01% = 0.000001%

This calculation assumes failures are independent—Server A failing doesn't make Server B more likely to fail.

**Reality: Correlated failures**
- Both servers share the same power substation
- Both servers use disks from the same manufacturing batch
- Both servers run identical software with the same bugs
- Both servers receive the same deployment at the same time

If a power substation fails, both servers go down simultaneously. If the disk batch has a defect that manifests after 18 months, both fail around the same time. If the software has a load-related bug, both hit it under the same traffic conditions. If a deployment has a problem, both servers receive it together.

The probability of "both servers failing" is no longer the product of independent probabilities—it's much higher, potentially approaching the single-server failure probability. Redundancy has provided minimal improvement.

## Common Infrastructure Causes of Correlated Failures

The most obvious sources of correlated failures involve shared physical infrastructure components.

### Physical Infrastructure Dependencies

**Power systems** create perhaps the most intuitive correlation risk. If multiple servers share the same power substation, electrical panel, UPS system, or generator, a failure in that shared component affects all servers simultaneously. A utility company performing substation maintenance, a lightning strike hitting a transformer, or a cooling failure in a UPS battery room can take down an entire data center worth of equipment.

This is precisely why AWS designs Availability Zones with separate power substations. Each AZ has independent power infrastructure—different utility feeds, different substations, different backup generators. A power event in one AZ doesn't cascade to another.

**Cooling infrastructure** represents a similar risk. Modern high-density servers generate tremendous heat—a single rack can draw 20-30 kilowatts and produce equivalent heat output. If HVAC systems fail, server temperatures rise rapidly. Most modern servers throttle performance as temperature increases and eventually shut down to prevent hardware damage. Shared cooling infrastructure means correlated thermal failures.

Again, Availability Zone design addresses this. Each AZ has independent cooling systems—separate chillers, different HVAC units, independent temperature monitoring and control systems. A cooling problem in one AZ is contained.

**Network connectivity** creates complex correlation risks. Servers in the same data center typically share upstream network infrastructure—router and switches in the aggregation layer, connections to internet service providers, physical fiber runs. A fiber cut, a misconfigured router, or a routing table error can simultaneously disconnect large numbers of servers.

AWS Availability Zones use diverse network paths. Each AZ connects to different upstream providers through different physical fiber routes. The interconnection between AZs uses dedicated metro fiber with multiple diverse paths. This reduces (though doesn't eliminate) network-related correlation.

### Hardware Component Correlation

**Batch manufacturing defects** can create subtle correlation. Server components—hard drives, memory modules, network cards, power supplies—are manufactured in batches. A defect affecting an entire batch might not manifest immediately. Instead, it appears after some number of hours of operation, or under specific stress conditions, or after exposure to certain temperature patterns.

If you purchase and deploy hundreds of servers at the same time, they likely contain components from the same production batches. When the defect manifests, it affects many servers simultaneously. The same server model ordered six months apart might have completely different components from different batches with different reliability characteristics.

**Component aging** creates time-based correlation. Hardware components have failure rates that increase with age. A fresh hard drive might have a 1% annual failure rate. After three years, that rate might be 5%. After five years, 10% or higher. If you deployed a fleet of 1000 servers all at once, their components age together, creating correlated increase in failure probability.

The failure curve for a batch of identical servers looks like a bell curve—few failures early, peak failures in the middle of life, fewer failures later (because most have already failed). If your entire fleet was deployed in a one-month window, they all hit that peak failure period together.

This is why large-scale operators use **continuous refresh** strategies, constantly replacing a percentage of servers with new hardware. Amazon deploys new servers continuously, gradually retiring old ones, maintaining a distribution of hardware ages. This spreads the peak failure period across time rather than concentrating it.

### Common Dependencies

**DNS represents a foundational dependency** that many systems share. If DNS becomes unavailable or returns incorrect results, applications can't resolve hostnames to IP addresses. Database connections fail. API calls to services fail. Even if your application servers are perfectly healthy, a DNS failure can make your entire service unavailable.

The classic joke among infrastructure engineers captures this: "It wasn't DNS. It definitely wasn't DNS! ...It was DNS." DNS problems are surprisingly common causes of outages, and because so many systems depend on DNS, these failures are highly correlated.

AWS addresses this through Route 53's architecture—distributed across many servers, using Anycast routing, with multiple levels of caching. But even with robust DNS infrastructure, application-level DNS caching, circuit breakers, and fallback mechanisms are prudent defenses against DNS-related correlation.

**Time synchronization** creates another subtle dependency. Many distributed systems depend on synchronized clocks for ordering events, coordinating actions, expiring credentials, or rotating logs. If NTP (Network Time Protocol) servers become unavailable or provide incorrect time, or if a sudden time adjustment occurs (like fixing a clock that was wrong), many systems can experience coordinated issues.

Certificate validation failures, coordinated cache expirations, scheduled job misfires, and distributed transaction coordination problems can all result from time-related issues. Because all servers in a fleet typically use the same NTP servers, time-related problems affect them simultaneously.

## Operational Practices That Create Correlation

Beyond physical infrastructure, operational practices can introduce correlation risks that are equally dangerous.

### Parallel Operations and Mass Actions

In the early days of operating distributed systems, it was common to execute commands across all servers in parallel. Need to grep for a specific log line? Run the command on all 1000 servers simultaneously. Need to adjust an operating system parameter? Use a configuration management tool to change it everywhere at once.

The efficiency is appealing—make changes quickly across the entire fleet. However, if anything goes wrong, it impacts every server simultaneously. Examples of problems that have caused production incidents:

**Resource exhaustion**: Running an expensive grep across logs on all servers simultaneously causes CPU spikes on every server, potentially degrading service or triggering cascading failures.

**Mistyped commands**: A typo in a mass-executed command (like accidentally including a space in a file path, or using the wrong flag) affects all servers identically.

**Unexpected side effects**: A seemingly safe parameter change has an unforeseen interaction with another setting, causing failures—but now it's failing everywhere at once.

Modern operational practices introduce **velocity controls** and **feedback loops** to make mass operations safer:

**Velocity controls** limit how quickly changes propagate. Instead of executing on all 1000 servers simultaneously:
- Execute on 1 server (the "one-box")
- Monitor metrics for problems
- Execute on 1% of servers (10 servers)
- Monitor metrics
- Execute on 10% of servers (100 servers)
- Monitor for any degradation
- Execute on remaining servers

If a problem occurs, it affects a small percentage of the fleet before you detect and halt the rollout. The blast radius is limited.

**Feedback loops** monitor for degradation during rollouts. Define key metrics (error rate, latency, availability) and automatically halt rollouts if these metrics show degradation beyond threshold. The combination of velocity controls and feedback loops transforms potentially catastrophic fleet-wide operations into low-risk, incrementally validated changes.

### Deployment Practices and Software Correlation

**Deploying software updates simultaneously to all servers** creates perhaps the most common source of operational correlation. Every server runs identical code, so a bug in new code affects all servers identically. A deployment problem (corrupted package, missing dependency, configuration incompatibility) affects all servers.

The solution is **phased rollouts** with clear stage gates:

```
Stage 1: One-box deployment
  - Deploy to single server
  - Monitor error rate, latency, resource utilization
  - Validate core functionality
  - Gate: No increase in errors, latency within bounds

Stage 2: 1% deployment
  - Deploy to 1% of fleet
  - Monitor for 30 minutes
  - Check detailed metrics and logs
  - Gate: Success rate ≥ baseline, latency ≤ baseline + 10%

Stage 3: 10% deployment
  - Deploy to 10% of fleet
  - Monitor for 1 hour
  - Run automated integration tests
  - Gate: All critical metrics within tolerance

Stage 4: 50% deployment
  - Deploy to half the fleet
  - Monitor for 2 hours
  - Check for unusual patterns
  - Gate: No increase in ticket volume, metrics stable

Stage 5: 100% deployment
  - Deploy to remaining servers
  - Monitor for 24 hours
  - Prepare rollback if needed
```

Each stage validates health before proceeding. Problems are detected when they affect a small percentage, giving time to roll back before impacting the entire fleet.

**Availability Zone awareness in deployments** adds another critical dimension. Even with phased rollouts, deploying to all Availability Zones simultaneously creates cross-zone correlation risk. If a deployment has a subtle zone-specific bug (perhaps triggered by specific network configurations or hardware characteristics), it could affect all zones at once.

AWS internal services follow **zonal deployment calendars**:

```
Monday: Deploy to use1-az1
  - Complete full phased rollout within the zone
  - Monitor for 24 hours

Wednesday: Deploy to use1-az2
  - Complete full phased rollout within the zone
  - Monitor for 24 hours

Friday: Deploy to use1-az3
  - Complete full phased rollout within the zone
  - Monitor for weekend

Following Monday: Deploy to remaining zones if applicable
```

This practice ensures that deployment problems affect only one Availability Zone at a time. Customers who architected applications across multiple zones experience minimal or zero impact. Problems are detected and rolled back in one zone while others continue running the previous, working version.

<Callout type="warning">
Never use operational tools that can simultaneously affect multiple Availability Zones without explicit safeguards. Even with velocity controls, a tool executing across all zones can undermine the isolation that makes multi-AZ architectures resilient. Always respect AZ boundaries in operational tooling.
</Callout>

## Regions and Availability Zones: Architected Isolation

AWS addresses infrastructure correlation risks through the fundamental design of Regions and Availability Zones. This architectural approach provides strong isolation guarantees that both AWS service teams and customers can leverage.

### Region-Level Isolation

Each **AWS Region** is completely isolated from other Regions at multiple levels:

**Independent infrastructure**: Regions don't share physical infrastructure. They have separate power grids (different utility companies), independent network providers (different ISPs and fiber carriers), and completely separate data centers. A massive power outage affecting the entire Eastern Seaboard might impact us-east-1, but us-west-2 continues operating normally.

**Independent control planes**: AWS services operate independent control planes in each Region. The EC2 control plane in us-east-1 shares no components with the EC2 control plane in eu-west-1. If there's a service impairment affecting EC2 in one Region, it doesn't propagate to other Regions.

**Independent deployment pipelines**: Software deployments to AWS services happen independently per Region. A bad deployment in one Region is detected and rolled back without affecting other Regions. This isolation ensures that software-related correlated failures are contained regionally.

**Network isolation**: Regions connect to each other through standard internet or AWS dedicated connections, but they don't share internal networking infrastructure. A regional network impairment stays within that Region.

### Availability Zone Design

Within each Region, **Availability Zones** add fine-grained isolation: physically separate facilities with independent power, cooling, and network, deployed to on independent calendars so software changes never affect multiple zones simultaneously, but connected over low-latency dedicated fiber so synchronous replication is practical. This design is covered in detail in [Blast Radius and Failure Boundaries](/course/fault-isolation/fault-isolation-boundaries).

For correlated-failure purposes, the relevant property is that AWS deliberately *engineers away* correlation across AZs — different substations, different fiber paths, different deployment days — so that an outage in one zone is highly unlikely to coincide with an outage in another.

## Software Uniformity and Behavioral Correlation

Even with perfect infrastructure independence, several correlation risks remain. The most significant: software uniformity and behavioral patterns.

### The Lemmings Problem: Identical Code, Identical Failures

In most distributed systems, every server runs identical software with identical resource limits and failure modes. Load balancers evenly distribute work across servers—exactly what we want for efficiency. However, this uniformity means that **any software bug or load-related failure affecting one server likely affects all servers simultaneously**.

This parallels a classic video game called Lemmings. Players guided creatures through landscapes filled with obstacles and deadly traps. Every lemming followed the same algorithm—when faced with an obstacle, turn around; when faced with a cliff or trap, walk straight ahead. Because every lemming followed identical logic, if one lemming encountered a trap and perished, all following lemmings met the same fate.

Distributed systems face similar challenges. Examples of software uniformity causing correlated failures:

**Resource exhaustion**: A specific request pattern causes memory leak. Because all servers run the same code, all servers develop the same leak under the same load conditions. Eventually, all servers exhaust memory around the same time.

**File descriptor limits**: A particular usage pattern causes the application to hold too many open file descriptors. In older systems, this was a common failure mode—one server would hit the limit, crash or hang, increased load would shift to other servers, causing them to hit the limit faster, creating a cascading failure.

**Connection pool saturation**: A specific database query holds connections longer than expected. All servers running this query eventually exhaust their connection pools simultaneously.

**CPU thrashing**: A specific code path is unexpectedly CPU-intensive. Under certain load conditions, all servers start using 100% CPU simultaneously, causing request timeouts across the entire fleet.

**Buffer overflow/underflow**: A specific input triggers a buffer handling bug. All servers receiving similar input trigger the same bug, potentially crashing or corrupting state.

Modern systems are less likely to run out of file descriptors (default limits are much higher), but other resource limits, behaviors, and bugs continue to create correlated failure modes. The uniformity of code means uniformity of failure.

### Techniques for Reducing Software Correlation

**Shuffle sharding** (covered in depth in a previous lesson) ensures no two customers see exactly the same combination of servers. By assigning each customer to a unique subset of the worker pool, problems affecting one customer's assigned servers don't affect other customers who use different workers.

**Gradual rollouts with feature flags** allow deploying new code to all servers while gradually enabling new features for subsets of users. If a new feature has a bug, it affects only the users for whom it's enabled, not the entire user base. This separates deployment from activation, providing another isolation boundary.

**Diversity through canary instances** runs a small percentage of the fleet on next week's code while most run current code. This provides early warning of problems in new code before it reaches the entire fleet. If canary instances show elevated errors or resource issues, the rollout halts.

**Canarying in production** involves running a small percentage of production traffic through new code paths even before deployment. Using feature flags, a small percentage of requests exercise new code while most use old code. This validates new code under real production load before widespread adoption.

### Correlated Behavioral Patterns

Even when systems don't fail, **correlated behaviors** can create problems. Consider a real-world scenario from the COVID-19 pandemic when many people worked from home:

During a home internet service provider (ISP) outage, many people working from home simultaneously switched to their cellular hotspots as backup connectivity. The cellular network in the neighborhood, unprepared for this surge, became overwhelmed and degraded. The ISP and cellular network were completely independent—different companies, different infrastructure, different technologies. Yet **correlated behavior** (everyone failing over at the same time) caused failure in the otherwise-unrelated backup system.

Similar risks exist in distributed systems where large numbers of clients change behavior simultaneously:

**Coordinated cache expiration**: All servers configured with the same TTL (time to live) for cached data simultaneously invalidate their caches and request fresh data from the database. This creates a massive coordinated load spike (often called a "thundering herd") that can overwhelm the database.

**Synchronized retry storms**: All clients failing a request implement retry logic with the same fixed delay (e.g., wait 5 seconds, retry). All clients retry simultaneously, creating a coordinated spike in load that prevents the system from recovering.

**Scheduled job stampedes**: All servers run housekeeping jobs (log rotation, cache cleanup, metrics aggregation) at the same time (midnight, top of each hour). This creates coordinated CPU and I/O spikes that degrade service availability.

**Coordinated credential refresh**: All services use credentials (API keys, OAuth tokens, IAM roles) that expire at the same time. All attempt to refresh credentials simultaneously, overwhelming the authentication service.

**Mode shift synchronization**: Distributed systems sometimes have mode shifts—transitioning from "normal operation" to "degraded mode," or from "cached state" to "refresh from source." If all servers use identical thresholds for these transitions, they all shift modes at the same time, creating coordinated behavior change.

## Using Jitter to Desynchronize Behaviors

**Jitter** — adding small amounts of randomness to common behaviors and limits — is the standard countermeasure for synchronized behavior. Apply it to:

- **Retry delays**: full or decorrelated jitter on exponential backoff so 1,000 simultaneously-failed clients don't all retry at the same instant
- **Cache TTLs**: stagger expirations across the fleet to prevent thundering-herd database queries when popular cache entries expire together
- **Periodic jobs and credential refresh**: scatter the work across the period so midnight cleanup or 1-hour-token refresh doesn't spike all at once
- **Resource limits**: vary connection-pool sizes or other limits slightly per host so the whole fleet doesn't hit identical limits at identical times

The detailed treatment with code samples for retry jitter (full jitter, decorrelated jitter, capped backoff) lives in [Retry Strategies That Don't Make Things Worse](/course/traffic-management/timeouts-retries-backoff).

### Make jitter deterministic for debuggability

Pure randomness makes behavior unreproducible. Seed the random value from a per-host identifier (hostname, instance ID) so each server consistently uses the same jitter values across runs, but different servers use different ones. You get fleet-wide desynchronization *and* reproducible per-host behavior — easier to debug when one host misbehaves.

## Cellular Architectures for Isolation

Beyond Availability Zones, many AWS services adopt **cellular architectures** to create additional isolation boundaries that limit correlation risks.

Instead of operating a single stack per Availability Zone or Region, a service is subdivided into a set of identical, independent partitions called **cells**. Each cell handles a subset of customers or requests and operates independently.

### Benefits of Cellular Architecture

**Blast radius limitation**: An unforeseen problem affects only one cell's customers. If you have 10 cells and one experiences an issue, only 10% of customers are impacted. Other cells continue operating normally.

**Simplified validation**: You can validate changes on a single cell before rolling to others. Deploy to Cell 1, monitor for issues, then deploy to remaining cells if successful. This provides finer-grained validation than one-box or percentage-based rollouts.

**Gradual rollout control**: Deploy to 1 cell (10% of traffic), then 3 cells (30%), then all cells (100%). This gives clear stage gates and easy rollback boundaries.

**Operational independence**: Cells can be managed independently—patched on different schedules, run different software versions during transitions, be taken offline for maintenance without affecting others.

**Correlated failure reduction**: Operational tooling and practices respect cell boundaries just as they respect AZ boundaries. Operations never affect multiple cells simultaneously, preventing operational correlation.

### Example Cellular Architecture

```
Region: us-east-1
├── Cell 1 (use1-az1, use1-az2, use1-az3)
│   ├── Load Balancer
│   ├── Application Servers (20 instances)
│   ├── Database Cluster
│   └── Serves: Customers A-F
│
├── Cell 2 (use1-az1, use1-az2, use1-az3)
│   ├── Load Balancer
│   ├── Application Servers (20 instances)
│   ├── Database Cluster
│   └── Serves: Customers G-M
│
└── Cell 3 (use1-az1, use1-az2, use1-az3)
    ├── Load Balancer
    ├── Application Servers (20 instances)
    ├── Database Cluster
    └── Serves: Customers N-Z
```

Each cell is a complete, independent copy of the service. Cells span Availability Zones for AZ-level isolation, and cell boundaries provide service-level isolation. A problem in Cell 1 (deployment issue, resource exhaustion, database problem) affects only Customers A-F. The other cells continue serving their customers normally.

**Routing to cells** typically happens at the ingress layer based on customer ID, tenant ID, or request characteristics. The routing decision determines which cell handles a request, and that cell provides complete service independently.

<KeyTakeaways>
- **Redundancy requires independence for effectiveness**: Multiple servers only improve availability if their failures are uncorrelated; correlated failures completely undermine the probabilistic benefits of redundancy
- **Infrastructure correlation sources**: Shared power/cooling/network, hardware from same production batches, component aging when fleet deployed simultaneously, and common dependencies like DNS create infrastructure-level correlation risks
- **Operational correlation sources**: Executing commands across all servers simultaneously, deploying software without phased rollouts, and lacking Availability Zone awareness in deployment practices create operational correlation that can cause fleet-wide failures
- **AWS Regions and AZs provide architected isolation**: Regions are completely independent (power, network, control planes, deployments); within Regions, Availability Zones use physical separation, independent infrastructure, and time-separated deployments to prevent correlated failures
- **Software uniformity creates behavioral correlation**: Identical code on all servers means identical bugs and resource limits; load distribution ensures problems affect all servers similarly under same conditions
- **Jitter prevents synchronized behaviors**: Adding small randomness to retry delays, cache TTLs, job schedules, credential expiration, and resource limits prevents thundering herds and coordinated behavior changes
- **Deterministic jitter maintains debuggability**: Seed random functions with server identifiers (hostname, IP) so each server uses consistent jitter values, enabling reproducible behavior while preventing fleet-wide synchronization
- **Cellular architectures add isolation layers**: Subdividing services into independent cells limits blast radius, simplifies validation, enables gradual rollouts, and provides operational independence beyond Availability Zone boundaries
</KeyTakeaways>
