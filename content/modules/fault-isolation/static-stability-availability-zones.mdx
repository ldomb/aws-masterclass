# Static Stability Using Availability Zones

## Introduction: Designing for Resilience Before Failures Happen

At Amazon, services must meet extraordinarily high availability targets. A 99.99% availability requirement means no more than 52 minutes of downtime per year. A 99.999% target allows just 5 minutes annually. These aren't aspirational goalsвҖ”they're contractual obligations backed by Service Level Agreements (SLAs) with financial penalties for violations.

Meeting such stringent targets requires rethinking how we approach dependencies. The traditional mindset treats dependencies as external factors we react to when they fail: "If this database becomes unavailable, we'll spin up a replacement." "If this server dies, we'll auto-scale to add capacity." This reactive approach has a fundamental flawвҖ”it relies on control plane operations (creating resources, modifying configurations, propagating changes) working correctly precisely when the overall system is already stressed or impaired.

The alternative is **static stability**вҖ”designing systems that continue operating in their current state without requiring changes when dependencies become impaired. A statically stable system might not see new data or updated information from an impaired dependency, but everything that was working before the impairment continues to work. There's no need to launch new instances, update routing, or reconfigure infrastructure. The system simply keeps running.

Availability Zones provide the foundational infrastructure building block that makes static stability practical at scale. When combined with static stability design principles, Availability Zones enable regional services to maintain operation through various failure scenarios without requiring complex failover orchestration or rapid capacity adjustments.

## Availability Zones as Fault Isolation Boundaries

This lesson assumes familiarity with how Availability Zones provide isolation вҖ” physical separation, independent power and network, and zonal deployment calendars that prevent correlated software failures. If you haven't already, read [Blast Radius and Failure Boundaries](/course/fault-isolation/fault-isolation-boundaries) for the full treatment.

The summary: each zone is engineered to fail independently, but zones in a Region connect over high-bandwidth dedicated fiber with single-digit-millisecond latency. That combination вҖ” independent enough to fail separately, close enough to replicate synchronously вҖ” is what makes multi-AZ architectures practical. Because each VPC subnet maps to exactly one AZ, subnet selection is implicitly zone selection.

The rest of this lesson focuses on what to *do* with those properties: design systems that keep working through AZ impairments without needing control-plane operations to recover.

## The control-plane / data-plane separation that enables static stability

Static stability is only possible because AWS services separate **control plane** (orchestration: launching, modifying, deleting resources) from **data plane** (the day-to-day work: forwarding packets, reading/writing objects, executing queries). The control plane is complex and lower-availability; the data plane is simpler and higher-availability. The two are covered in depth in [Blast Radius and Failure Boundaries](/course/fault-isolation/fault-isolation-boundaries).

The relevant property for this lesson: **the EC2 data plane is designed to be statically stable in the face of control plane impairments**. The hypervisor has local copies of route tables, security group rules, and ARP entries. If the EC2 control plane goes down, existing instances keep routing packets, reading EBS volumes, and serving traffic вҖ” they just can't be launched, modified, or terminated. The same separation applies to S3, DynamoDB, RDS, ELB, Lambda, and most other AWS services.

This is the foundation everything else in this lesson builds on.

## Static Stability Patterns for High Availability

Understanding static stability principles enables us to design better multi-AZ architectures. The key insight: **prepare for impairments before they happen** rather than planning to react after they occur.

The less effective approach: Deploy to multiple Availability Zones expecting that if one zone becomes impaired, you'll scale up (using Auto Scaling or manual intervention) in other zones to restore capacity. This requires successful control plane operations during the impairmentвҖ”creating new instances, updating load balancer targets, modifying DNS records.

The statically stable approach: Overprovision infrastructure so that even if an entire Availability Zone becomes impaired, the system continues operating correctly without launching any new instances or making any configuration changes.

### Pattern 1: Active-Active for Horizontally Scalable Services

For stateless, horizontally scalable workloads, deploy fleets across three or more Availability Zones with deliberate overprovisioning. The goal is having enough capacity that losing an entire zone doesn't require any control plane operations.

**Capacity calculation for three zones**: If your service requires six instances during normal operation to handle peak load at acceptable performance, deploy three instances in each of three zonesвҖ”nine total instances. Each zone operates at approximately 66% of its tested capacity limit.

When an Availability Zone fails, the remaining two zones have six instancesвҖ”exactly the capacity you need. No new instances need to launch. No configuration changes are required. Load balancers automatically stop sending traffic to the failed zone's instances (they fail health checks), and the remaining instances absorb the load.

**The mathematics**: With three Availability Zones, overprovision by 50% (9 instances instead of 6). With four zones, overprovision by 33% (8 instances instead of 6). The formula: For n zones, maintain (n/(n-1)) Г— baseline_capacity.

If minimum count is too low, Auto Scaling might scale in during quiet periods, removing the buffer you need when a zone fails. Configure scaling policies carefully to avoid this trap.

Amazon RDS with Multi-AZ deployment automatically creates this architecture. You provide a DB subnet group (a collection of subnets spanning multiple AZs), and RDS places the primary in one zone and the standby in a different zone.

**How standby failover provides static stability:**

During normal operation:
- Applications connect to the database using a DNS name like `mydb.cluster-xxx.us-east-1.rds.amazonaws.com`
- This DNS name points to the primary instance in AZ-1
- All writes and reads go to the primary
- The standby in AZ-2 receives synchronous replication of all writes
- The standby is "hot"вҖ”fully configured, has current data, ready to serve traffic

When AZ-1 becomes impaired:
- RDS detects the failure through health checks and network monitoring
- RDS automatically promotes the standby to primary
- RDS updates the DNS record to point to the new primary (in AZ-2)
- Applications continue using the same connection string
- DNS TTL is short (typically 30 seconds), so applications pick up the new endpoint quickly
- Total failover time: typically 60-120 seconds

**No infrastructure changes required**: The standby instance was already running, already had current data, already was configured correctly. The only change is DNS, which is a data plane operation (Route 53's data plane has high availability). No control plane operations neededвҖ”no launching new instances, no provisioning EBS volumes, no configuring security groups.

If the leader in AZ-1 fails (due to AZ impairment or instance failure), the followers in AZ-2 and AZ-3 detect missing heartbeats and initiate a new leader election. **Static stability in both patterns**: The commonality is that all capacity needed during a zone failure was provisioned before any failure occurred. Neither pattern requires deliberate control plane dependencies (launching instances, modifying infrastructure) to respond to Availability Zone impairments.

## Availability Zone Independence: Advanced Internal Patterns

Let's examine how AWS applies Availability Zone independence internally, particularly in foundational services like Amazon EC2. These patterns are most relevant when building services that need to be highly available themselves while also providing infrastructure for others to be highly available.

### Zonal Deployment Calendar in Practice

EC2 follows strict zonal deployment practices for all componentsвҖ”physical servers hosting instances, edge devices managing network traffic, DNS resolvers, control plane components in the launch path, and operational tools.

**Deployment schedule example**:
```
Week 1:
  Monday: Deploy to use1-az1 (one-box, then 1%, then full zone)
  Wednesday: Deploy to use1-az2 (one-box, then 1%, then full zone)
  Friday: Deploy to use1-az3 (one-box, then 1%, then full zone)

Week 2:
  Monday: Deploy to use1-az4 (if exists)
  Continue to other zones in region
```

Each zone's deployment includes the standard phased rollout: deploy to one server first, validate functionality, deploy to 1% of servers, monitor metrics, deploy to 10%, continue monitoring, etc. But crucially, **this entire phased rollout completes in one zone before beginning in the next**.

If a deployment contains a bug that causes failures, it affects only the zone currently being deployed to. Engineers detect the issue through automated alarms or canary metrics, halt the rollout, roll back the problematic deployment, and investigateвҖ”all while other zones continue running the previous version normally.

Customers who deployed across multiple zones experience minimal or zero impact. Their instances in other zones continue operating on working software. If they properly implemented health checks and multi-AZ redundancy, their applications continue serving traffic without degradation.

### When to Use Zonal Architecture

**Use zonal architecture when**:
- The service is foundational infrastructure that others depend on for AZ independence
- The service is part of a data plane that must have high availability
- The service sits in critical paths where AZ impairments must not spread
- Network traffic costs benefit from staying local (AWS charges for cross-AZ data transfer)

Examples: NAT Gateway, VPC endpoints, load balancer nodes, DNS resolvers, network appliances.

**NAT Gateway as an example**: EC2's NAT Gateway is a zonal service. Customers instantiate a NAT Gateway separately in each Availability Zone and configure route tables to use the local NAT Gateway for outbound internet traffic.

## Data Durability Across Availability Zones

While operational components should often stay within Availability Zones (for the independence benefits discussed above), data durability requires a different approach. **Hard state must be replicated across multiple zones** for disaster recovery.

This might seem contradictoryвҖ”we want AZ independence for operations but AZ distribution for data. The distinction is critical:

**For data plane operations** (reading/writing data, processing requests, routing traffic):
- Keep operations local to AZ when possible
- Avoid cross-AZ dependencies in critical paths
- Enable true zone independence

**For data durability** (ensuring data survives AZ failures):
- Replicate across multiple AZs
- Accept cross-AZ replication as necessary overhead
- Ensure business-critical data exists in multiple physical locations

### Typical Replication Patterns

**RDS automated backups**:
- Periodic snapshots stored in S3 (which automatically replicates across AZs)
- Point-in-time recovery logs stored in S3
- Even if primary and standby instances both fail, restore from backup

**Read replicas for disaster recovery**:
- Maintain read replicas in different AZs
- Replicas can be promoted if primary fails
- Asynchronous replication provides recovery point (slight data lag acceptable)

**Application-level state replication**:
- Distributed systems often maintain state in multiple replicas
- Consensus protocols (Raft, Paxos) ensure consistency
- Place replicas in different AZs for durability

**S3 automatic replication**:
- S3 automatically replicates data across multiple AZs in a region
- Every object exists in at least three different AZs
- Provides 99.999999999% (11 nines) durability

**Important principle**: These replicas aren't necessary for the primary zone to function during normal operation. They exist solely for recovery purposes. If AZ-1 is working normally, the replicas in AZ-2 and AZ-3 could be unavailable without affecting primary operations. But if AZ-1 fails, those replicas enable recovery.

This contrasts with active-active data plane operations, where all zones must work for full capacity. Data replication is insurance, not operational dependency.

## The Cost-Benefit Analysis of Static Stability

Static stability requires overprovisioningвҖ”running more infrastructure than needed during normal operation. Is this cost justified?

### The Cost Side

**50% additional compute for three-AZ deployment**:
- Baseline need: 6 instances
- Static stability deployment: 9 instances
- Additional cost: 3 instances = 50% increase

**Storage overhead for standby databases**:
- Primary database: 1TB storage
- Standby replica: 1TB storage (synchronized)
- Additional cost: 1TB = 100% increase (but only for storage, not compute)

**Network costs for data replication**:
- Cross-AZ data transfer for synchronous replication
- Ongoing cost based on write volume
- Typically small compared to compute costs

### The Benefit Side

**Eliminated control plane dependencies during failures**:
- No need to launch instances when zone fails
- No need to update load balancer configurations
- No need to modify DNS records
- No dependency on control plane availability during impairment

**Faster recovery time**:
- Active-active: Instant (automatic load balancer failover)
- Active-standby: 60-120 seconds (database failover)
- vs. Traditional scaling: 5-10 minutes (launch instances, wait for healthy)

**Reduced operational complexity**:
- No complex failover orchestration
- No multi-step recovery procedures
- No need to test control plane operations under stress

**Higher availability outcomes**:
- Fewer components needed for recovery = fewer failure modes
- Simpler recovery path = higher success probability
- Immediate capacity = shorter customer impact

**Better compliance with SLA targets**:
- 99.99% availability = ~52 minutes downtime/year
- Saving 5 minutes per incident = more incidents tolerable
- Static stability pattern reduces downtime per incident

For services with stringent availability requirements, the cost of overprovisioning is small compared to the benefit of higher availability and simpler operations. This is especially true when considering the true cost of outages: SLA penalties, customer dissatisfaction, revenue loss, and reputation damage.

<KeyTakeaways>
- **Static stability eliminates reactive dependencies**: Systems designed to continue operating without changes during dependency failures are more reliable than systems requiring control plane operations (launching instances, updating configs) to recover
- **Control planes vs data planes have different roles**: Control planes orchestrate changes (complex, many dependencies, lower availability target); data planes maintain operations (simpler, local state, higher availability target); separating them enables data plane static stability
- **Active-active pattern for stateless services**: Deploy across 3+ AZs with 50% overprovisioning so losing one zone doesn't require launching new capacity; each zone operates at ~66% of tested capacity limit
- **Active-standby pattern for stateful services**: Pre-provision standby resources (database replicas, cluster followers) in different zones; failover promotes existing standby rather than creating new resources
- **Availability Zones provide strong isolation**: Physical separation (up to 60 miles), independent infrastructure (power, cooling, network), time-separated deployments, and zone-local traffic flows prevent correlated failures
- **Zonal vs regional architecture**: Foundational services keep traffic within zones (2/3 probability of avoiding impaired zone regardless of call chain length) while higher-level services use simpler regional-calls-regional pattern ((2/3)^N probability)
- **Data durability requires cross-AZ replication**: While operations stay local for independence, hard state must replicate across zones for disaster recovery; use RDS automated backups, read replicas, or application-level replication
- **Cost-benefit favors static stability for critical services**: 50% compute overprovisioning is justified by eliminating control plane dependencies, reducing recovery time from minutes to seconds, and achieving higher availability SLAs
</KeyTakeaways>
