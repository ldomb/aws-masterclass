## Introduction

One of Amazon's guiding tenets for building solutions is simple yet profound: **Avoid walking through one-way doors**. This means staying away from choices that are hard to reverse or extend. While this principle applies across all stages of software development—from designing products and APIs to architecting backend systems—it's particularly critical for deployments.

A deployment transitions software from one state (version) to another. Both states may work perfectly, but the transition itself—whether forward (upgrade or roll-forward) or backward (downgrade or rollback)—can cause the software to malfunction. When this happens, it leads to service disruptions that make systems unreliable for customers.

The challenge isn't ensuring both versions work independently; it's ensuring the transitions between them are safe. At Amazon, teams fully prepare for rollback before every deployment. They plan and verify that software is **backwards compatible** at every revision—meaning it can be rolled back without errors or disruption to functionality available in the previous version.

This isn't just good practice; it's a fundamental requirement for operating reliable distributed systems at scale. Before releasing a new version of software, Amazon tests it in beta or gamma test environments along multiple dimensions such as functionality, concurrency, performance, scale, and downstream failure handling. This testing helps uncover problems in the new version and fix them.

However, it may not always be sufficient to ensure a successful deployment. Teams may encounter unexpected circumstances or suboptimal software behavior in production environments. Amazon wants to avoid putting teams in a situation where rolling back the deployment could cause errors for customers. To avoid being in this situation, teams fully prepare themselves for a rollback before every deployment.

<Callout type="info">
A version of software that can be rolled back without errors or disruption to the functionality available in the previous version is called **backwards compatible**. Amazon plans and verifies that software is backwards compatible at every revision.
</Callout>

## Standalone vs. Distributed Deployments

Understanding rollback safety begins with recognizing the fundamental differences between standalone and distributed software deployments. The complexity and challenges increase dramatically when moving from standalone applications to distributed systems.

### Standalone Software: Atomic Transitions

For standalone software running as one process on one device, deployments are **atomic**—two versions never run simultaneously. The deployment process is straightforward: stop the old version, start the new version. At no point do both versions coexist.

If the software maintains state (persisting data to disk or memory), the requirement for safe transitions is straightforward:
- The new version must **read (deserialize)** data written (serialized) by the old version
- The old version must **read (deserialize)** data written (serialized) by the new version

Satisfying this bidirectional compatibility condition makes deployment safe for rolling both forward and backward. If version 2 can read version 1's data and version 1 can read version 2's data, you can deploy version 2 and roll back to version 1 safely.

### Distributed Systems: The Complexity of Coexistence

In distributed systems, deployments become significantly more complex. Deployments use **rolling updates** to maintain availability—the new version rolls out to a subset of hosts while others continue serving requests. These hosts typically communicate through:
- **Remote procedure calls (RPC)** for synchronous communication
- **Shared persistent state** (metadata, checkpoints, queues) for asynchronous communication

This communication presents additional challenges that don't exist in standalone systems:

1. **Version Coexistence**: During rolling deployments, old and new versions run side-by-side, potentially for hours
2. **Mixed Communication**: The writer and reader could be running different versions of software
3. **Data Interpretation**: Different versions could interpret the same data differently
4. **Failure Modes**: The reader may fail to read the data altogether, causing an outage during the deployment itself—not just during a rollback

<Callout type="warning">
In distributed systems, rollback safety must account for version coexistence. During rolling deployments, old and new versions run side-by-side. Both forward transitions (upgrades) and backward transitions (rollbacks) must be safe with mixed-version fleets.
</Callout>

### The Rolling Update Challenge

Consider a distributed service running on 100 hosts. During a rolling update deploying 10 hosts at a time:

| Time | Hosts with Old Version | Hosts with New Version | Mixed Fleet Duration |
|------|----------------------|----------------------|---------------------|
| T0 | 100 | 0 | - |
| T1 | 90 | 10 | Started |
| T2 | 80 | 20 | Ongoing |
| ... | ... | ... | ... |
| T10 | 0 | 100 | Ended |

For potentially hours during this deployment, both versions are in production, communicating with each other. Any incompatibility between versions will manifest as production issues affecting real customers.

## Problems with Protocol Changes

The most common reason for being unable to roll back safely is a **change of protocol**—the format, structure, or semantics of how services communicate or store data. These changes can be obvious or surprisingly subtle.

### Obvious Protocol Changes: Data Format Modifications

Consider a code change that starts compressing data while persisting it to disk. The implementation seems straightforward:

**Before (Version 1):**
```python
def write_data(data):
    disk.write(data)

def read_data():
    return disk.read()
```

**After (Version 2):**
```python
def write_data(data):
    compressed = compress(data)
    disk.write(compressed)

def read_data():
    compressed = disk.read()
    return decompress(compressed)
```

After the new version writes compressed data, **rolling back isn't an option**—the old version doesn't know it needs to decompress data after reading from disk. The deployment has created a **one-way door**.

The problem manifests in multiple ways:
- **During deployment**: If data is stored in a blob or document store, servers running the old version will fail to read compressed data written by servers running the new version
- **After deployment**: All new data is compressed, making rollback impossible without data conversion
- **Between services**: If compressed data passes between processes or servers, receivers running the old version will fail to read it

### Subtle Protocol Changes: Timing and Heartbeats

Sometimes protocol changes are extremely subtle and can be overlooked during code review. Consider two servers communicating asynchronously over a connection. To keep each other aware they're alive, they send **heartbeats** every 5 seconds. If a server doesn't see a heartbeat within the stipulated time, it assumes the other server is down and closes the connection.

Now consider deploying a change that increases the heartbeat period from 5 to 10 seconds. The code commit seems minor—just a number change:

```python
# Before
HEARTBEAT_INTERVAL = 5

# After
HEARTBEAT_INTERVAL = 10
```

However, it's now **unsafe to both roll forward and backward**:

**During Forward Deployment:**
- Server running new version (v2) sends heartbeats every 10 seconds
- Server running old version (v1) expects heartbeats every 5 seconds
- After 5 seconds without a heartbeat, v1 server terminates the connection
- In a large fleet, this happens with several connections
- Result: Availability drop during deployment

**During Rollback:**
- The same problem occurs in reverse
- Servers rolling back to v1 expect 5-second heartbeats
- Servers still on v2 send 10-second heartbeats
- Connections are terminated, causing another availability drop

Such subtle changes are hard to analyze by reading code or design documents. Therefore, Amazon explicitly verifies that each deployment is safe for rolling forward and backward through testing.

### Other Common Protocol Change Examples

**Message Queue Format Changes:**
```python
# Before: Messages as simple strings
queue.publish("user_id:12345")

# After: Messages as JSON
queue.publish(json.dumps({"user_id": 12345, "timestamp": now()}))

# Problem: Old consumers can't parse JSON, new consumers expect JSON
```

**API Response Structure Changes:**
```python
# Before
return {"status": "success", "data": [...]}

# After: Added metadata field
return {"status": "success", "data": [...], "metadata": {...}}

# Problem: If old clients do strict validation, they reject the extra field
```

**Database Schema Evolution:**
```sql
-- Before: Single name field
CREATE TABLE users (id INT, name VARCHAR(100));

-- After: Split into first and last name
CREATE TABLE users (id INT, first_name VARCHAR(50), last_name VARCHAR(50));

-- Problem: Old code looks for 'name' field that no longer exists
```

## Two-Phase Deployment Technique

One way Amazon ensures safe rollback is using the **two-phase deployment technique**. This approach divides potentially breaking changes into two separate, independently safe deployments.

### The Scenario: XML to JSON Migration

Consider a service managing data on Amazon S3 that runs on a fleet of servers across multiple Availability Zones. Currently, the service uses **XML format** to persist data—all servers write and read XML. For business reasons (better tooling, smaller payloads, easier parsing), the team wants to switch to **JSON format**.

If made in one deployment:
- Servers that pick up the change will write JSON
- Other servers don't yet know how to read JSON
- **Result**: Errors during deployment

Therefore, the change divides into two phases that can be deployed days or weeks apart.

### Phase 1: Prepare (Teach Reading New Format)

In the **Prepare** phase, deploy version V2 that prepares all servers to **read JSON** (in addition to XML) but continues to **write XML**.

**Code Changes in V2:**
```python
def write_data(data):
    # Still write XML
    xml_data = convert_to_xml(data)
    s3.put_object(Body=xml_data)

def read_data(key):
    raw_data = s3.get_object(Key=key)['Body'].read()

    # NEW: Try JSON first, fallback to XML
    try:
        return json.loads(raw_data)
    except json.JSONDecodeError:
        return parse_xml(raw_data)
```

**Operational Impact**: None
- All servers can still read XML
- All data is still written in XML
- System behavior is identical to V1 from a functional perspective

**Rollback Safety**: High
- If we roll back this change (V2 → V1), servers revert to a condition where they can't read JSON
- This is fine because **no data has been written in JSON yet**
- No customer impact from rollback

    B -.->|Safe rollback<br/>no JSON written| A
    style A fill:#e1f5ff,color:#1a1a1a

### Phase 2: Activate (Switch to Writing New Format)

In the **Activate** phase, deploy version V3 that activates servers to **write in JSON format** while maintaining the ability to read both formats.

**Code Changes in V3:**
```python
def write_data(data):
    # NEW: Write JSON instead of XML
    json_data = json.dumps(data)
    s3.put_object(Body=json_data)

def read_data(key):
    raw_data = s3.get_object(Key=key)['Body'].read()

    # Keep reading both formats
    try:
        return json.loads(raw_data)
    except json.JSONDecodeError:
        return parse_xml(raw_data)
```

**During Deployment**:
- As each server picks up V3, it starts writing JSON
- Servers that haven't picked up V3 yet still write XML
- All servers (both V2 and V3) can read both formats
- **Result**: No errors, safe mixed-version operation

**Rollback Safety**: High
- If we roll back (V3 → V2), all data written by servers temporarily in V3 is in JSON
- Data written by servers not yet in V3 is in XML
- This is fine because V2 servers can read **both formats**
- No customer impact from rollback

### General Applicability

Although this example shows serialization format change from XML to JSON, the general technique applies to **all protocol changes**:

**Heartbeat Example:**
- **Prepare Phase**: All servers relax expected heartbeat period to 10 seconds but continue sending heartbeats every 5 seconds
- **Activate Phase**: All servers change sending frequency to 10 seconds

**Database Schema Example:**
- **Prepare Phase**: Add new columns (first_name, last_name) but keep writing to old column (name) and copy to new columns
- **Activate Phase**: Start writing only to new columns

**API Versioning Example:**
- **Prepare Phase**: API accepts both old and new request formats
- **Activate Phase**: Clients start sending new format

## Precautions with Two-Phase Deployments

Amazon takes specific precautions when following two-phase deployment to ensure safety and success. These precautions have been learned through years of experience with large-scale distributed systems.

### Precaution 1: Verify Complete Fleet Upgrade

Many deployment tools consider deployment successful if a **minimum number of hosts** pick up the change and report as healthy. For example, AWS CodeDeploy has a deployment configuration called `minimumHealthyHosts` that might be set to 75% or 90%.

**The Critical Assumption**: At the end of the first phase, **all servers** (100%, not 90% or 95%) have been upgraded to read both formats.

**Why This Matters**: If one or more servers fail to upgrade during the Prepare phase:
- They would fail to read data written in the new format during the Activate phase
- This causes production errors affecting customer requests routed to those servers
- The impact may be subtle—affecting only a percentage of traffic—making it hard to diagnose

<Callout type="tip">
**Best Practice**: Explicitly verify that all servers have picked up the change in the Prepare phase. Don't rely on minimum healthy host configurations—verify 100% deployment success before proceeding to the Activate phase.
</Callout>

**Amazon DynamoDB Example**: When working on Amazon DynamoDB, engineers coordinated protocol communication changes between massive numbers of servers spanning multiple microservices. They:
1. Coordinated deployments so **all servers** reached the Prepare phase first
2. Then proceeded to the Activate phase
3. As a precaution, explicitly verified deployment succeeded on **every single server** at each phase end
4. Used automated tooling to check server versions and deployment status

### Precaution 2: Allow Bake Period Between Phases

While each of the two phases is safe for rolling back individually, **you cannot roll back both changes together**.

**The Problem**: At the end of the Activate phase (V3):
- Servers write data in JSON
- The software version before both Prepare and Activate (V1) doesn't know how to read JSON
- Rolling back directly from V3 to V1 causes all new data to be unreadable

<Mermaid title="Safe vs Unsafe Rollback Paths During Format Migration" chart={`graph LR

    B --> C[V3 Activate<br/>Write JSON<br/>Read XML+JSON]
    C -.->|Safe| B
    B -.->|Safe| A
    C -.->|UNSAFE<br/>JSON unreadable| A
    style A fill:#e1f5ff,color:#1a1a1a
    style B fill:#FFE4B5,color:#1a1a1a
    style C fill:#90EE90,color:#1a1a1a

**Best Practice**: Let considerable time pass between Prepare and Activate phases. This **bake period** usually lasts **a few days to weeks**:

| Phase | Bake Time | Purpose |
|-------|-----------|---------|
| After Prepare | 3-7 days | Ensure no need to roll back to pre-Prepare version |
| After Activate | 7-14 days | Build confidence in new format, prepare for backfilling |

During the bake period:
- Monitor metrics carefully for any degradation
- Watch for any unusual error patterns
- Ensure sufficient data points at production load
- Gain confidence that the Prepare phase is stable

This ensures teams don't need to roll back to the version before Prepare, making the Activate phase safe to deploy.

### Precaution 3: Plan for Old Format Removal (Backfilling)

After the Activate phase, you **can't safely remove** the software's ability to read the old format immediately. It's unsafe because all data written before the Prepare phase is in the old format.

You can only remove the ability to read XML after ensuring **every single object** has been rewritten in JSON. This process is called **backfilling**.

**Backfilling Strategies**:

**1. Passive Backfilling (Lazy Migration)**:
```python
def read_data(key):
    raw_data = s3.get_object(Key=key)['Body'].read()

    try:
        data = json.loads(raw_data)
        format_type = 'json'
    except json.JSONDecodeError:
        data = parse_xml(raw_data)
        format_type = 'xml'

    # If we read XML, rewrite as JSON
    if format_type == 'xml':
        write_data(key, data)  # Rewrites as JSON

    return data
```

**2. Active Backfilling (Proactive Migration)**:
- Run a separate background job that iterates through all objects
- Read each object and rewrite it (triggering conversion to new format)
- Track progress to know when migration is complete
- Can be throttled to avoid impacting production load

**Backfilling Challenges**:
- May require additional tooling that runs concurrently while the service writes and reads data
- Can take **weeks or months** for large data sets (billions of objects)
- Must be throttled to avoid overwhelming the system
- Requires tracking progress and ensuring completeness
- Need to handle objects that change during backfilling

Only after backfilling is 100% complete can teams safely deploy V4 that removes the old format reading code:

```python
def read_data(key):
    raw_data = s3.get_object(Key=key)['Body'].read()
    return json.loads(raw_data)  # Only JSON now
```

## Best Practices for Serialization

Most software involves serializing data—whether for persistence or network transfer. As software evolves, serialization logic commonly changes. Amazon has developed best practices through years of experience managing these changes at scale.

### 1. Avoid Custom Serialization Formats

**The Temptation**: Custom serialization logic may seem trivial initially and can provide better performance. It's tempting to write:

```python
# Custom format: pipe-separated values
def serialize(user):
    return f"{user.id}|{user.name}|{user.email}"

def deserialize(data):
    parts = data.split('|')
    return User(id=parts[0], name=parts[1], email=parts[2])
```

**The Problem**: Subsequent iterations present challenges already solved by established frameworks:
- What if a user's name contains a pipe character?
- How do you add optional fields?
- How do you handle null values?
- How do you version the format?
- How do you deprecate fields?

**Best Practice**: Use established frameworks like:
- **JSON**: Human-readable, widely supported, flexible schema
- **Protocol Buffers**: Efficient binary format, strong typing, schema evolution
- **Cap'n Proto**: Zero-copy serialization, schema evolution
- **FlatBuffers**: Direct memory access, no parsing needed

When used appropriately, these frameworks provide safety features like:
- **Escaping**: Special characters handled automatically
- **Backwards compatibility**: Old code can read new data
- **Forward compatibility**: New code can read old data
- **Attribute existence tracking**: Distinguish explicitly set fields from default values

### 2. Explicitly Version Serializers

With each change, explicitly assign a **distinct version** to serializers independent of source code or build versioning.

```python
class DataSerializer:
    VERSION = 3  # Increment with each format change

    def serialize(self, data):
        payload = {
            'version': self.VERSION,
            'data': data,
            'timestamp': time.time()
        }
        return json.dumps(payload)

    def deserialize(self, raw_data):
        payload = json.loads(raw_data)
        version = payload.get('version', 1)  # Default to v1 for old data

        if version == 1:
            return self._deserialize_v1(payload)
        elif version == 2:
            return self._deserialize_v2(payload)
        elif version == 3:
            return self._deserialize_v3(payload)
        else:
            raise ValueError(f"Unknown version: {version}")
```

**Store the serializer version** with the serialized data or in metadata:
- In the data itself (as shown above)
- In file metadata
- In database columns
- In message headers

**Older serializer versions continue to function** in new software. Never remove the ability to read old versions until backfilling is complete.

<Callout type="info">
**Metrics for Visibility**: It's usually helpful to emit metrics for the version of data written or read. This provides operators with visibility and troubleshooting information if errors occur. This applies to RPC and API versions too.
</Callout>

### 3. Avoid Serializing Uncontrolled Data Structures

Don't serialize data structures you can't control. The internal implementation may change unexpectedly.

**Bad Example**:
```java
// Serializing Java's HashMap using reflection
HashMap<String, Object> data = new HashMap<>();
data.put("user_id", 12345);
data.put("scores", new TreeSet<>(Arrays.asList(95, 87, 92)));

// Serialize using reflection
byte[] serialized = SerializationUtils.serialize(data);
```

**Problems**:
- When upgrading the JDK, underlying implementation of HashMap or TreeSet may change
- Deserialization could fail with ClassNotFoundException or InvalidClassException
- This risk applies to classes from libraries shared across teams
- You have no control over internal implementation changes

**Better Example**:
```java
// Define your own serializable DTO
public class UserData {
    public int userId;
    public List<Integer> scores;

    // Explicit control over serialization
}

UserData data = new UserData();
data.userId = 12345;
data.scores = Arrays.asList(95, 87, 92);

String json = objectMapper.writeValueAsString(data);
```

### 4. Allow Unknown Attributes

Typically, design serializers to **allow the presence of unknown attributes**. Where feasible, serializers should **retain unknown attributes** while writing back data.

**Implementation Example**:
```python
class FlexibleDataStore:
    def __init__(self):
        self.known_fields = {'user_id', 'name', 'email'}
        self.data = {}
        self.unknown_fields = {}

    def deserialize(self, raw_data):
        all_data = json.loads(raw_data)

        for key, value in all_data.items():
            if key in self.known_fields:
                self.data[key] = value
            else:
                # Preserve unknown fields
                self.unknown_fields[key] = value

    def serialize(self):
        # Include both known and unknown fields
        all_data = {**self.data, **self.unknown_fields}
        return json.dumps(all_data)
```

**Benefits**: With this accommodation:
- Even if a server running the new software version includes new attributes while serializing
- Servers running the old version won't wipe out those attributes while updating the same data
- **This can eliminate the need for two-phase deployment** in some scenarios

**Example Scenario**:
1. V2 server writes data with new field: `{"user_id": 123, "name": "Alice", "premium": true}`
2. V1 server reads data, preserves unknown `premium` field, updates name: `{"user_id": 123, "name": "Alice Smith", "premium": true}`
3. V2 server reads data, sees `premium` field intact

## Verifying Rollback Safety Through Testing

Amazon explicitly verifies that software changes are safe to roll forward and backward through **upgrade-downgrade testing**. This requires setting up test environments representative of production.

### Test Environment Requirements

Over the years, Amazon has identified patterns to **avoid** in test environment setup:

#### Anti-Pattern 1: Single-Server Services

**Problem**: Test environments where each service has only one server make all deployments atomic, precluding the possibility of running different versions concurrently.

**Why It Matters**: This masks the very problems two-phase deployment solves—version coexistence issues that only appear when old and new versions run side-by-side.

**Best Practice**: Even if test environments don't see production-level traffic, use:
- Multiple servers from different Availability Zones behind each service
- Same deployment configuration as production
- Load balancers distributing traffic across servers
- Just as in production

#### Anti-Pattern 2: All-at-Once Deployments

**Problem**: Test environments with multiple servers but deploying to all at once also prevent old and new versions from running simultaneously.

**Why It Matters**: Again, this masks version coexistence issues. The deployment succeeds in test but fails in production.

**Best Practice**: Use the same deployment configuration in test and production environments:
- Same rolling deployment batch sizes
- Same deployment strategies (blue/green, canary, rolling)
- Same health check configurations

### Deployment Order Considerations

For changes involving coordination between microservices, maintain the same deployment order across microservices in test and production. However, forward and backward order could differ.

For serialization changes, Amazon generally follows this order:

| Direction | Order | Rationale |
|-----------|-------|-----------|
| **Rolling Forward** | Readers before writers | Readers must understand new format before writers produce it |
| **Rolling Backward** | Writers before readers | Stop producing new format before removing ability to read it |

**Example: Three-Service Architecture**

Services: A (producer) → B (processor) → C (consumer)

**Forward Deployment (XML to JSON)**:
1. Deploy C (consumer) - can now read JSON
2. Deploy B (processor) - can now read and write JSON
3. Deploy A (producer) - starts producing JSON

**Backward Rollback**:
1. Rollback A (producer) - stops producing JSON
2. Rollback B (processor) - stops processing in JSON
3. Rollback C (consumer) - no longer needs to read JSON

This appropriate ordering is consistently followed in test and production environments.

### The Upgrade-Downgrade Testing Process

When test environment setup mirrors production:

**1. Simulate Production Traffic**:
- Create and read records in quick succession
- Exercise all APIs continuously
- Generate realistic load patterns
- Mix of read and write operations

**2. Progress Through Three Stages**, each lasting long enough for all APIs, workflows, and batch jobs to run at least once:

**Stage 1 - Partial Deployment (Version Coexistence)**:
- Deploy the change to approximately half the fleet
- Monitor for errors or unexpected behavior
- Verify both versions can communicate
- Check shared data stores for compatibility issues
- Duration: Long enough for all code paths to execute

**Stage 2 - Complete Deployment**:
- Complete the deployment to all servers
- All servers now running new version
- Monitor for any issues that only appear at 100% new version
- Duration: Again, long enough for full system exercising

**Stage 3 - Rollback Deployment**:
- Initiate rollback and follow the same steps
- Roll back to approximately half the fleet first
- Complete rollback until all servers run old software
- Monitor for the same compatibility issues
- Verify no data corruption or loss

**3. Evaluate Results**:
If there are no errors or unexpected behavior during these stages, consider the test successful.

**Metrics to Monitor**:
- Error rates (4xx, 5xx responses)
- Latency percentiles (P50, P90, P99)
- Success rates for all APIs
- Database query performance
- Queue depths and processing rates
- Memory and CPU utilization

<Callout type="warning">
**Failure is Valuable**: If upgrade-downgrade testing reveals incompatibilities, you've discovered them in test rather than production. This is exactly why you do this testing. Fix the issues and test again.
</Callout>

## Real-World Example: DynamoDB Protocol Change

A real-world example from Amazon DynamoDB illustrates the scale and coordination required for two-phase deployments in large distributed systems.

**The Change**: DynamoDB needed to change the communication protocol between massive numbers of servers spanning multiple microservices.

**The Challenge**:
- Thousands of servers across multiple microservices
- High-throughput, low-latency requirements
- Global deployment across all AWS Regions
- Cannot afford any downtime or data loss

**The Solution - Two-Phase Deployment**:

**Phase 1 - Prepare (3 weeks)**:
- **Week 1**: Deploy to beta environment, verify thorough testing
- **Week 2**: Deploy to gamma and internal production environments
- **Week 3**: Begin global production deployment
- **Verification**: Explicitly verified deployment succeeded on every single server
- **Coordination**: Synchronized across all microservices in the system
- **Result**: All servers could read both old and new protocol

**Bake Period (1 week)**:
- Monitor all metrics for any anomalies
- Ensure no need to roll back to pre-Prepare version
- Build confidence in the stability of the change

**Phase 2 - Activate (3 weeks)**:
- **Same gradual rollout** as Phase 1
- Servers start using new protocol for writes
- Mixed protocol operation during rollout
- **Monitoring**: Intensive monitoring of all communication patterns
- **Result**: Smooth transition with zero customer impact

**Total Timeline**: 7 weeks from start to finish for a critical protocol change affecting thousands of servers.

**Key Success Factors**:
- Rigorous upgrade-downgrade testing in all environments
- Explicit verification of 100% deployment success
- Careful coordination across multiple teams and microservices
- Conservative bake period between phases
- Extensive monitoring and rollback readiness

<KeyTakeaways>
- **Protocol changes are the most common rollback hazard**: Changes to data formats, RPC protocols, timing parameters, or any communication mechanism can make rollbacks unsafe—always evaluate transitions, not just individual versions
- **Two-phase deployment enables safe protocol changes**: The Prepare phase makes all servers compatible with both old and new protocols, while the Activate phase switches to the new protocol, ensuring rollback safety at each step
- **Verify complete fleet upgrade between phases**: Explicitly confirm 100% of servers successfully deployed the Prepare phase before proceeding to Activate—partial deployments can cause widespread failures
- **Bake time between phases is critical**: Allow days or weeks between Prepare and Activate phases to ensure no need to roll back past the Prepare phase, which would make Activate changes unsafe
- **Use established serialization frameworks**: Avoid custom formats in favor of proven frameworks like JSON or Protocol Buffers that provide built-in backward compatibility, forward compatibility, and safety features
- **Backfilling requires planning**: After switching to a new format, plan for months of backfilling old data before removing the ability to read old formats
- **Test both upgrade and downgrade paths**: Upgrade-downgrade testing in production-like environments with multi-server deployments catches rollback issues that code review and design analysis miss
- **Deployment order matters for multi-service changes**: Deploy readers before writers when rolling forward, writers before readers when rolling backward to maintain compatibility throughout transitions
</KeyTakeaways>
