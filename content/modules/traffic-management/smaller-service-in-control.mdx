## Introduction

At Amazon, we build large-scale distributed systems composed of smaller, independent services, each with a narrow set of responsibilities. This architectural approach—decomposing complex systems into focused, manageable components—is fundamental to how AWS operates. These services interact with each other over well-defined APIs, enabling teams to scale, evolve, and operate each one independently. A team can deploy a new version of their service without coordinating with dozens of other teams. They can scale their service based on its specific load patterns. They can reason about their service's behavior without understanding every detail of every other service.

This decomposition brings enormous benefits, but it also introduces a subtle, dangerous challenge that many engineers encounter only after their systems reach significant scale. The challenge emerges from a seemingly simple question: **How should two services of vastly different sizes communicate with each other?**

Within AWS, a common architectural pattern splits systems into two distinct components:

**Data plane**: Services responsible for executing customer requests—the actual work of computing, storing, retrieving, and processing data. These must scale to handle all customer workload, so data plane fleets are large, sometimes containing tens of thousands of servers.

**Control plane**: Services responsible for managing and vending customer configuration—coordinating system behavior, maintaining operational metadata, handling administrative operations like creating resources, updating settings, and managing lifecycles. Since configuration changes happen far less frequently than data operations, control plane fleets are much smaller, sometimes just hundreds of servers.

In this lesson, we discuss different ways the data plane and control plane can interact with each other to avoid system overload. In many architectures, the natural instinct is to have the larger data plane fleet call the smaller control plane fleet to fetch configuration or push state. This seems straightforward and is simpler to implement initially.

However, we also want to share the remarkable success Amazon has had when we **put the smaller fleet in control**—having the smaller control plane dictate the pace of interactions rather than letting the much larger data plane overwhelm it with requests.

## The Standard Pattern: Data Plane Calls Control Plane

Let's examine Amazon EC2 as a concrete example. The data plane consists of physical servers where customers' EC2 instances run—millions of virtual machines processing customer workloads. The control plane consists of services that interact with the data plane, performing critical coordination functions:

- Telling each server about the EC2 instances it needs to run
- Keeping running instances up to date with Amazon VPC configuration changes
- Receiving metering data, logs, and metrics emitted by the servers
- Deploying new software versions to the servers
- Managing instance lifecycle operations like launches, terminations, and migrations

Although the actual names and responsibilities vary across similar architectures, these systems share two critical characteristics:

1. **The data plane and control plane need to stay synchronized**: The data plane needs to receive configuration updates from the control plane, and the control plane needs to receive operational state from the data plane
2. **The size of the data plane fleet exceeds the size of the control plane fleet**, frequently by a factor of 100 or more—sometimes by a factor of 1000 or more

When building such architectures at Amazon, one of the decisions we carefully consider is the **direction of API calls**. Should the larger data plane fleet call the control plane fleet? Or should the smaller control plane fleet call the data plane fleet?

### The Simplest Approach

For many systems, having the data plane fleet call the control plane tends to be the simpler of the two approaches initially. In such systems, the control plane exposes APIs that can be used to:

- Retrieve configuration updates (data plane pulls configuration)
- Push operational state (data plane pushes metrics, logs, status)

The data plane servers call those APIs either periodically (polling every N seconds) or on-demand (when events occur that require fetching new configuration or pushing updates).

Each data plane server initiates the API request, so **the control plane does not need to keep track of the data plane fleet**. For each request, a control plane server responds with updated configuration (usually by querying its durable data store like DynamoDB), and then forgets about it. Stateless request handling is simple and well-understood.

To provide fault tolerance and horizontal scalability, control plane servers are placed behind a load balancer like Application Load Balancer. The load balancer distributes incoming requests across available control plane servers.

This is one of the simpler types of distributed systems to build, and it has inherent **availability advantages** through its simplicity:

- **No fleet tracking**: The control plane doesn't need to maintain a registry of all data plane servers
- **Stateless handling**: Each request is independent; no complex state management
- **Horizontal scalability**: Add more control plane servers behind the load balancer as needed
- **Well-understood patterns**: Standard request/response patterns that most engineers know

However, at Amazon we have learned that when the scale of the data plane fleet exceeds the scale of the control plane fleet by a factor of 100 or more, this type of distributed system requires **careful fine-tuning to avoid the risk of overload**.

### The Steady State vs. The Unsteady State

In the steady state, the larger data plane fleet periodically calls control plane APIs, with requests from individual servers arriving at **uncorrelated times**. Server 1 might poll every 10 seconds starting at time 0. Server 2 might poll every 10 seconds starting at time 0.347. Server 3 might poll every 10 seconds starting at time 2.891. The poll times are spread out across the interval.

The load on the control plane is evenly distributed over time, and the system operates smoothly. Everything hums along nicely.

However, unexpected shifts in the calling pattern of the data plane can increase the load on the control plane dramatically—to the point of completely overwhelming it. Several situations can cause such shifts:

**Code or configuration bugs in the data plane** that increase the frequency of API calls. Perhaps a bug causes servers to poll every 1 second instead of every 10 seconds—a 10× increase in request rate.

**Recovery from an outage** where all data plane servers try to call control plane APIs at the same time. Imagine the data plane fleet reboots after a software update. All servers come online within a short time window, and they all immediately call the control plane to fetch configuration. Instead of requests spread evenly over 10 seconds, thousands of requests arrive simultaneously.

**A small number of failed API calls that cause retries**, which causes increased load on the control plane, which causes even more failures and more retries, creating a feedback loop. This is the dreaded retry storm or cascade failure.

By examining these situations, we've observed that what they have in common is a **small change in the environment that causes the clients to act in a correlated manner and start making concurrent requests at the same time**.

Without careful fine-tuning, this shift in behavior can overwhelm the smaller control plane. This change in the volume of concurrent requests can cause the load on the system to increase beyond its **breaking point**, after which its goodput (the amount of useful work the system performs) quickly drops to zero.

Once goodput drops to zero, the system is effectively down. Control plane servers are consuming maximum CPU, network, and other resources, but producing no useful results. Requests time out. Data plane servers retry, adding even more load. The system is trapped in a metastable failure state.

### Preventing Overload Through Careful Tuning

To avoid overloading the system, both the control plane and the data plane need to be carefully tuned to ensure the load on the control plane never exceeds the breaking point.

**On the control plane**, mechanisms like load shedding can help the system continue to make progress even in the face of unexpected load shifts. Load shedding intentionally discards some requests to keep goodput as high as possible rather than letting the system thrash and accomplish nothing.

**On the data plane**, carefully tuning the frequency of requests, and adding backoff and jitter to retries can help reduce the risk of correlated increases in request rates. Jitter ensures that even if all servers experience a triggering event simultaneously, their subsequent retry times are spread out rather than synchronized.

These techniques work and Amazon uses them extensively. But they require constant vigilance and tuning. As the scale mismatch grows—as the data plane fleet grows from 100× larger to 1000× larger than the control plane—the tuning becomes increasingly delicate and the risks increasingly severe.

## Solving Scale Mismatch with Amazon S3

The biggest challenge with the standard pattern is the **scale mismatch** itself. The control plane fleet is badly outnumbered by the data plane fleet. At Amazon, we looked to our storage services for a solution. When it comes to serving content at massive scale, the service we commonly use is Amazon S3.

Instead of exposing APIs directly to the data plane, the control plane can periodically write updated configuration into an **Amazon S3 bucket**. Data plane servers then poll this bucket for updated configuration and cache it locally.

Similarly, to stay up to date on the data plane's operational state, the control plane can poll an Amazon S3 bucket into which data plane servers periodically write that information.

This architecture offers several compelling advantages:

**Simple to implement**: Reading and writing files to S3 is straightforward. Most programming languages have well-tested S3 clients. There's no complex protocol to implement.

**Amazon S3 is scaled to support even the largest client fleets**: S3 is one of the largest distributed systems in the world, designed to handle astronomical request rates. Even a data plane fleet of 100,000 servers polling every few seconds is easily handled.

**The control plane fleet can stay relatively small**: As the size of the data plane fleet grows, the control plane fleet size doesn't need to grow proportionally. The control plane just writes to S3; S3 handles serving the configuration to the data plane.

**Static stability**: If the control plane has an outage, the data plane can continue running with the last known configuration, even as servers come in or out of service. This property—called **static stability**—is a highly desirable attribute in distributed systems. The system continues functioning even when dependencies are unavailable.

Because of these significant advantages, this architecture is a popular choice at Amazon.

### AWS Hyperplane: A Real-World Example

An excellent example of a system using this architecture is **AWS Hyperplane**, the internal Amazon network function virtualization system that powers multiple AWS services including:

- Network Load Balancer
- NAT Gateway
- AWS PrivateLink

The AWS Hyperplane data plane contains devices that process customer traffic—millions of packets per second requiring microsecond-level latency. These devices need to know about the configuration of individual Network Load Balancers, NAT Gateways, and PrivateLink connections.

Here's how it works:

1. A periodic task within the AWS Hyperplane control plane scans its Amazon DynamoDB tables containing customer configuration
2. It writes that configuration into several Amazon S3 files
3. The data plane periodically downloads these files (every few seconds)
4. The data plane uses the file content to update internal routing configuration

This simple, robust pattern elegantly solves the scale mismatch problem. The control plane doesn't care how large the data plane fleet grows—it just writes files to S3. S3 handles distributing those files to millions of servers if needed.

## When S3-Based Architecture Doesn't Work

Despite its popularity and advantages, this architecture doesn't work in all cases. Two primary limitations drive Amazon to seek alternative approaches:

### Large Amount of Dynamic Configuration

In systems with a large amount of dynamic configuration that changes constantly and at large scale, it might be **impractical to continuously recompute the entire configuration and write it into Amazon S3**.

An example is Amazon EC2, where:
- A large number of data plane servers need to know about constantly changing configuration
- EC2 instances are being launched and terminated continuously
- Instance assignments to servers change
- VPC configurations update
- IAM roles change
- EBS volumes attach and detach

Recomputing and writing the entire EC2 configuration to S3 every few seconds would be prohibitively expensive computationally and would create configuration files of enormous size.

### Low-Latency Propagation Requirements

In other systems, it's important for changes in the control plane configuration to be **reflected in the data plane in single-digit seconds or faster**.

An example is a container service that notifies a server about a new container it needs to run. When a customer launches a new container, they expect it to start within seconds, not tens of seconds. Polling periodically updated files in Amazon S3 can make it impossible to achieve acceptable propagation latencies.

The polling interval creates a fundamental lower bound on propagation latency. If data plane servers poll every 10 seconds, configuration changes take up to 10 seconds to propagate (average case: 5 seconds). Making the polling interval shorter increases request rates and costs.

<Callout type="warning">
**The Polling Dilemma**: Shorter polling intervals mean faster propagation but higher request rates and costs. Longer polling intervals mean lower costs but slower propagation. For some workloads, no polling interval achieves acceptable latency without unacceptable cost.
</Callout>

When faced with scenarios involving dynamic configuration or low-latency requirements, Amazon looks for other approaches where **the small fleet can be in control of the pace** at which requests flow through the system.

## Inversion of Scale: Control Plane Pushes to Data Plane

One alternative approach reverses the flow of API calls entirely: have the smaller control plane fleet **push configuration changes** to the larger data plane fleet.

An architecture in which the smaller control plane fleet dictates the pace of work is **more resilient when overloaded**. Here's why:

If the control plane is overloaded or underscaled, it still continues making progress, even though at a slower pace than desired. The control plane processes as many push operations per second as it can handle. Configuration updates take longer to propagate, but they do eventually propagate. The system degrades gracefully.

In contrast, when the data plane fleet calls the control plane and the control plane is overloaded, goodput can drop to zero. The control plane becomes completely unresponsive. No configuration updates propagate at all until the overload situation is resolved.

However, this push-based approach also comes with several significant downsides that make it challenging to implement:

### Challenge 1: Fleet Inventory Management

**The control plane needs to keep an up-to-date inventory of all available data plane servers** so that it can push configuration updates to each one.

This inventory must be maintained continuously:
- When new data plane servers are added (auto-scaling up)
- When servers are removed (auto-scaling down, failures, deployments)
- When servers become temporarily unreachable (network issues)

Maintaining this inventory correctly is non-trivial and introduces complexity and potential failure modes in the control plane itself.

### Challenge 2: Handling Unreachable Servers

**At any point in time, some data plane servers will be unreachable**, and the control plane needs to deal with that gracefully.

Servers might be unreachable for many reasons:
- Rebooting for software updates
- Experiencing temporary network partitions
- Undergoing maintenance
- Failed and awaiting replacement

The control plane must track which servers successfully received configuration updates and which didn't. It must implement retry logic with appropriate backoff. It must avoid retry storms that could overwhelm recovering servers.

### Challenge 3: Work Distribution

**The control plane needs mechanisms to ensure that every data plane server receives configuration updates from at least one control plane server, and that no control plane server is responsible for too many data plane servers.**

This is particularly challenging as control plane servers come in and out of service (deployments, auto-scaling, failures). The distribution of responsibility must rebalance continuously without creating gaps in coverage or excessive overlaps.

### Using Consistent Hashing for Work Distribution

Amazon often uses **consistent hashing** to solve the work distribution challenge. This approach, which bears similarities to leader election, works as follows:

**1. Heartbeats**: Each control plane server sends a heartbeat into a shared data store (like DynamoDB) to indicate it's alive and available. The heartbeat includes a timestamp.

**2. Discovery**: Independently, each control plane server scans the data store for a list of other control plane servers with recent heartbeats (within the last N seconds). This gives each server a view of the current control plane fleet.

**3. Consistent Hashing**: Each control plane server deterministically executes a consistent hashing algorithm to identify a subset of data plane servers it's responsible for.

Because all control plane servers run the same algorithm on the same inputs (the list of live control plane servers), they reach consistent conclusions about responsibility without explicit coordination. If a control plane server fails, its heartbeats stop, other servers detect this, and they automatically redistribute responsibility for its data plane servers.

This approach is elegant but adds significant implementation complexity compared to the simple stateless request/response pattern.

## Decoupling Discovery from Control Flow

To avoid the three major complications of push-based architectures (fleet inventory, handling unreachable servers, work distribution) while maintaining their overload resilience benefits, Amazon often **decouples the direction of discovery from the direction of the control flow**.

Here's how it works:

Instead of control plane servers initiating connections to data plane servers, **each server in the larger data plane fleet opens a long-lived connection to a single control plane server**. This connection establishment can be done through a load balancer like Network Load Balancer, so data plane servers don't need to know the specific control plane server addresses.

After the connection is established, **the control plane server takes over and begins making API calls over that connection**, similar to the HTTP server push mechanism or WebSockets.

This architecture provides compelling benefits:

**Control plane dictates the pace**: Because the control plane server makes API calls over the connection it accepted, it controls how quickly configuration updates are pushed. This provides the same overload resilience as the pure push model—if the control plane is busy, it just pushes updates more slowly. Goodput never drops to zero.

**No fleet inventory required**: The control plane doesn't need to maintain a registry of data plane servers. Data plane servers establish connections, and the control plane accepts them. If a server goes away, its connection closes, and the control plane forgets about it. If a new server is added, it establishes a connection, and the control plane handles it.

**No distributed algorithms**: No consistent hashing or complex coordination logic is needed. All decisions are made using only local state on each control plane server. This dramatically reduces implementation complexity.

**Automatic retry handling**: If a control plane server is unavailable or busy, it can **reject the connection**. The data plane server then tries a different control plane server (the load balancer routes to a different backend). The data plane server will also try a different control plane server if the connection is terminated for any reason.

The trade-offs:

**More sophisticated communication protocol**: Instead of simple request/response HTTP, you need a protocol that supports bidirectional communication or server push. This could be WebSockets, HTTP/2 server push, gRPC streaming, or a custom protocol.

**Connection management overhead**: Long-lived connections need to be monitored with heartbeats or keepalives. Connection state must be managed. Connection limits must be considered.

Despite these trade-offs, Amazon has found this pattern to be extraordinarily successful for systems where the data plane significantly outnumbers the control plane.

## Extreme Scale Mismatch: UDP Notification Pattern

When the data plane outnumbers the control plane by a factor of **1000:1 or more**, even connection establishment can overload the control plane. The cost of performing TCP and TLS handshakes becomes prohibitive when thousands of clients try to connect simultaneously.

Amazon's solution for extreme scale mismatch: instead of data plane servers initiating TCP connections, they send small **UDP requests**. The control plane receives these lightweight UDP notifications and then **initiates the connection back to the data plane server**.

Why UDP?

**Handling UDP requests costs significantly less than TCP/TLS handshakes**: A UDP request is a single packet that can be processed in microseconds. A TCP handshake requires a three-way exchange (SYN, SYN-ACK, ACK). A TLS handshake requires additional round trips for certificate exchange and encryption negotiation. These add up to milliseconds per connection.

When 10,000 data plane servers all try to establish connections simultaneously, the difference between handling 10,000 UDP packets and handling 10,000 TCP+TLS handshakes is the difference between a few seconds and complete overload.

**UDP notifications let the control plane control the rate of connection establishment**: Rather than accepting connections as fast as they arrive (potentially overwhelming itself), the control plane can establish connections at its own pace. It processes UDP notifications, adds them to a work queue, and establishes connections at a sustainable rate.

A variation of this approach is used by **Amazon EC2 control planes** to push configuration updates to the Nitro system—the underlying hypervisor and hardware platform for EC2 instances. Given the astronomical scale of EC2, this pattern is essential for maintaining stability.

**Pattern Selection Guide**:
- **S3-based**: Use for eventually-consistent configuration, simple implementation, static stability
- **Push or reverse-connection**: Use for dynamic, low-latency requirements where the data plane is 100-1000× larger than control plane
- **UDP notification + reverse-connection**: Use for extreme scale mismatches (1000:1+) where connection establishment overhead becomes prohibitive
</Callout>

## Architectural Trade-Offs Summary

Let's compare the architectural approaches across several dimensions:

| Dimension | Data Plane Polls Control Plane | S3-Based | Control Plane Pushes | Reverse Connection | UDP Notification |
|-----------|-------------------------------|----------|---------------------|-------------------|------------------|
| **Implementation Complexity** | Low | Very Low | High | Medium | Medium-High |
| **Scale Limit** | ~100:1 with tuning | Unlimited | ~1000:1 | ~1000:1 | ~10000:1+ |
| **Propagation Latency** | Polling interval | Polling interval | Immediate | Immediate | Immediate |
| **Overload Resilience** | Requires tuning | Excellent | Excellent | Excellent | Excellent |
| **Static Stability** | No | Yes | No | No | No |
| **Fleet Inventory Needed** | No | No | Yes | No | No |
| **Connection Overhead** | Request/response | Request/response | Push connections | Long-lived | UDP + long-lived |
| **When to Use** | Small scale mismatch | Eventually-consistent config | Dynamic config, medium scale | Dynamic config, large scale | Extreme scale |

## Real-World Impact: Design Lessons

Amazon's experience with these patterns has shaped fundamental design principles:

**1. Recognize scale mismatch early**: If clients outnumber a service by 100:1 or more during design phase, plan for correlated behavior from the start. Don't wait until production to discover the overload risk.

**2. Smaller service controls pace**: When scale mismatch is significant, architect so the smaller fleet dictates the rate of interaction. This prevents catastrophic overload that drops goodput to zero.

**3. Avoid correlation**: If the larger fleet must call the smaller fleet, use jitter aggressively. Ensure clients don't act in lockstep. Stagger polling. Randomize retry times. Break synchronization.

**4. S3 is a powerful intermediary**: For many use cases, using S3 between control plane and data plane elegantly solves the scale mismatch while providing static stability. Don't overlook this simple solution.

**5. Test overload scenarios**: Fine-tuning is only effective if you know the breaking point. Test what happens when all clients act in lockstep. Measure where goodput drops to zero. Design defenses around those measurements.

**6. Gradual degradation beats sudden failure**: Architectures where the smaller service controls pace degrade gradually under overload (slower propagation) rather than failing suddenly (zero goodput). This is far preferable operationally.

## Conclusion

At Amazon, we've learned that if we don't pay attention to the relative scale of a service and its clients, distributed systems can be at serious risk of overload. In such situations, a subtle change in the behavior of clients—a bug, a mass reboot, a retry storm—can increase the load on the service to the point of overwhelming it completely.

Such changes in behavior can be hard to predict, and protecting against them requires careful fine-tuning of both the service and its clients. This fine-tuning is fragile and needs continuous maintenance as the system evolves.

At Amazon, we pay close attention to situations where clients outnumber a service by a factor of 100 or more, and we look for approaches where **the smaller fleet can control the pace** at which changes flow through the system.

The simplest approach uses a large-scale service like **Amazon S3 as an intermediary** between the small fleet and its clients. S3 is designed for astronomical scale and provides excellent static stability properties. For many systems, this is the right choice.

When that's not practical—because configuration is highly dynamic or propagation latency must be very low—we look to **inversion of scale**, an approach where the smaller fleet calls the larger one, dictating the pace of updates.

The most elegant implementation of inversion of scale **decouples discovery from control flow**: the larger fleet establishes connections, but the smaller fleet controls what happens over those connections. This provides overload resilience without requiring complex fleet inventory or distributed coordination algorithms.

For extreme scale mismatches (1000:1 or greater), **UDP notifications** allow the smaller fleet to control even the rate of connection establishment, preventing connection storms from overwhelming the system before configuration updates can even begin.

Although these approaches can add implementation complexity compared to simple request/response patterns, we find that they make it far easier to scale and operate the smaller fleet, even as the larger fleet continues to grow in size. The operational benefits—gradual degradation instead of sudden failure, no risk of goodput dropping to zero, no need for constant fine-tuning—justify the additional design and implementation effort.

When designing your own systems, consider the scale mismatch between components. If one component outnumbers another by 100:1 or more, the simple patterns may not suffice. Put the smaller service in control, and your system will be far more resilient to the unexpected correlated behaviors that inevitably emerge in large-scale distributed systems.

<KeyTakeaways>
- **Scale mismatch creates overload risk**: When clients outnumber a service by 100:1 or more, subtle environmental changes (bugs, reboots, retries) can cause correlated client behavior that overwhelms the smaller fleet, driving goodput to zero
- **S3-based architectures elegantly solve many cases**: Using Amazon S3 as an intermediary between small control planes and large data planes provides proven unlimited scale, static stability, and operational simplicity for eventually-consistent configuration
- **Inversion of scale provides overload resilience**: Having the smaller fleet control the interaction pace prevents catastrophic overload—if the small fleet is busy, updates propagate more slowly but never drop to zero goodput
- **Decouple discovery from control flow**: The architectural pattern where larger fleets establish connections but smaller fleets control communication over those connections combines the benefits of push (overload resilience) without the complexity (no fleet inventory, no distributed algorithms)
- **UDP notifications handle extreme scale**: When scale mismatch exceeds 1000:1, lightweight UDP requests let the control plane establish connections at its own pace, preventing connection storms from overwhelming the system before any work can begin
- **Static stability is valuable**: S3-based patterns let data planes continue operating with last-known configuration even when control planes are unavailable, providing a critical resilience property for foundational infrastructure
- **Gradual degradation beats sudden failure**: Architectures where smaller services control pace degrade gracefully under overload (slower propagation) rather than failing catastrophically (zero goodput), dramatically improving operational experience
- **Fine-tuning is fragile**: Relying on careful tuning of request rates and retry logic to prevent overload in asymmetrically-scaled systems is fragile and requires constant maintenance—architectural solutions (S3, reverse connections, UDP) are more robust
</KeyTakeaways>
