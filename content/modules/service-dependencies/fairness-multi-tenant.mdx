## Introduction

Multi-tenant systems are the foundation of cloud computing's efficiency and cost-effectiveness. By serving multiple customers or workloads from shared infrastructure, services like DynamoDB, Lambda, and API Gateway achieve higher resource utilization, lower operational costs, and reduced environmental impact compared to single-tenant architectures.

However, multi-tenancy introduces a fundamental challenge: **fairness**. How do you ensure that one customer's unpredictable spike in traffic doesn't degrade performance for other customers? How do you prevent a runaway workload from consuming all available resources? How do you make a multi-tenant system feel like a single-tenant system to each customer?

The answer lies in sophisticated **admission control systems** that implement API rate limiting—also called throttling or quota enforcement. These systems shape incoming traffic in different ways, such as prioritizing client workloads that remain within their planned usage, while applying backpressure to workloads that spike unpredictably.

This lesson explores how Amazon builds fairness into multi-tenant systems through rate-based quotas, load shedding, token bucket algorithms, and architectural patterns that help customers avoid hitting limits in the first place. Understanding these patterns is essential for both building cloud services and using them effectively, as fairness mechanisms directly impact application availability and performance.

The goal of fairness in multi-tenant systems is simple yet profound: **every client should experience what feels like a single-tenant system, even though they're sharing infrastructure with thousands of other clients**.

## The Evolution from Single-Tenant to Multi-Tenant Databases

The journey from single-tenant to multi-tenant architecture illustrates why fairness is so critical. To understand this evolution, let's explore David Yanacek's experience working on Amazon's infrastructure in the mid-2000s.

### The Single-Tenant Database Problem

Early in his career at Amazon, David worked on a team responsible for automating the operations of the Amazon.com web server fleet. His team built systems that:
- Forecasted how many servers to provision over time for each website fleet
- Monitored the health of servers
- Performed automatic remediation and replacement of broken servers
- Helped teams deploy software to the website

To manage state for many of these tools, the team needed databases. At Amazon today, the default choice would be a NoSQL database because these databases are designed for manageability, scalability, availability, and predictable performance. However, this was before those technologies were widely available, so the team set up servers running **MySQL with replication pairs** for high availability and redundancy.

They implemented backup and restore, tested for failure, and tried to ensure they could rely on this otherwise single point of failure.

### The Temptation to Share

When the team built new services and tools that needed new databases, they were often tempted to re-use existing databases by simply adding more tables to them. Why was this temptation so strong?

**First, utilization was low.** Looking at database server metrics showed that the servers were not heavily loaded. These particular databases were at the scale of:
- The number of developers deploying code to the website (hundreds, not thousands)
- The number of web servers they had at the time (thousands, not millions)

**Second, operational overhead was painful.** Configuring new databases, monitoring them, upgrading them, and trying to automate every aspect of their operation required significant effort. The team was small, and every new database meant more work.

So they created ad-hoc multi-tenant databases by sharing database servers across multiple applications:

### The Inevitable Failure

When the team gave in to the temptation to use the same set of database servers for multiple applications, they regretted it.

David remembers being on call and getting paged for a performance degradation in the deployment tool. He scratched his head trying to figure out what was wrong with the deployment service itself. Then he realized that a completely separate tool that the team owned (a fleet auditor of some kind) was running its **nightly state synchronization cron job**. This put a lot of extra load on the shared database.

**The fleet auditor tool didn't mind that the database was slow**—it was a batch job that could take hours. **But the deployment tool (and its users) sure did!** Developers trying to deploy their code were getting timeouts and errors because a completely unrelated batch job was consuming all the database capacity.

This is a classic **noisy neighbor problem** in multi-tenant systems. One tenant's workload impacts another tenant's experience, even though they should be independent.

### The Kobayashi Maru Scenario

This constant tension created what felt like a no-win scenario—a **Kobayashi Maru** (from Star Trek):

**Option 1: Share databases**
- Lower infrastructure costs (one server instead of many)
- Lower operational costs for some tasks (one backup/restore process)
- BUT: Terrible workload isolation, unpredictable performance, operational nightmares

**Option 2: Separate databases**
- Better workload isolation
- Predictable performance per application
- BUT: Higher infrastructure costs, much higher operational overhead, underutilized resources

The types of databases Amazon used were **meant for a single tenant**, so unsurprisingly, the team ran into pain when they tried to use them in a multi-tenant way. Traditional relational databases provide no built-in mechanisms for:
- Limiting query complexity per application
- Throttling requests per tenant
- Isolating I/O or memory usage by workload
- Automatic scaling based on individual tenant needs

### Amazon RDS: Automation but Not True Multi-Tenancy

When Amazon Relational Database Service (RDS) was launched, it made operations easier by automating much of the operational work:
- Automated backups and point-in-time recovery
- Automated patching and upgrades
- Monitoring and metrics
- High availability with Multi-AZ deployments
- Read replicas for scaling read traffic

This made it easier to run single-tenant systems as separate databases rather than sharing the same database across multiple applications. However, some workloads were quite small and others varied in size, so teams still needed to pay attention to the utilization of each database to get the instance sizes just right. Plus, they needed enough headroom to handle periodic swings in load.

RDS improved the operational experience but didn't fundamentally solve the multi-tenancy problem.

### DynamoDB: Purpose-Built for Multi-Tenancy

Later at Amazon, when David was seeking a new challenge, he learned about a new type of database being built at AWS. The goals of the database were to be:
- Highly elastic (scale up and down automatically)
- Scalable (handle any size workload)
- Available (no single points of failure)
- Low latency (single-digit millisecond response times)
- Fully managed (zero operational overhead for customers)

These goals were compelling because as a software engineer, David really disliked doing the same thing over and over again, especially if it's hard to do. So he expended great effort trying to automate repeatable tasks. This new database seemed like the perfect opportunity to finally fully automate every aspect of database maintenance that he found painful.

He joined the team that launched **Amazon DynamoDB** in 2012. Just like Captain Kirk, they used programming to pass the Kobayashi Maru test!

<Callout type="info">
DynamoDB was designed from day one as a multi-tenant system with fairness built in. Unlike RDS, where you provision entire database instances, with DynamoDB you simply interact through APIs, and the service handles all the multi-tenancy complexity behind the scenes.
</Callout>

### DynamoDB's Multi-Tenant Architecture

DynamoDB leverages multitenancy to provide a database that is highly elastic, durable, and available. Unlike when using Amazon RDS, when you create resources in DynamoDB, you don't provision a whole Amazon EC2 instance. You simply interact with your database through an API, and behind the scenes DynamoDB figures out the **fraction of a server** that's required for the workload.

Actually, it's using a fraction of **multiple servers in multiple Availability Zones** for high availability and durability. As your workload grows and shrinks, DynamoDB adjusts that fraction, and enlists more servers or fewer, as needed.

This is true multi-tenancy: thousands of customers sharing the same physical infrastructure, but each experiencing what feels like their own dedicated database with predictable performance.

## The Case for Multitenancy

Service-oriented architecture (SoA) is a core part of the Amazon culture of strong ownership and loose coupling of teams and systems. This architecture comes with another important benefit: **improved hardware efficiency through resource sharing**.

### How Multi-Tenant Onboarding Works

If another application wants to make use of an existing service, the service owner doesn't need to do much work to take on a new tenant. After reviewing the use case and performing a security review, the service owner grants the new client system access to call particular APIs or access particular data.

**No additional infrastructure setup or installation needs to happen** for the service owner to create and operate a new copy of the service for this new use case—it just re-uses the existing one.

This is fundamentally different from traditional enterprise software, where onboarding a new customer often meant:
- Provisioning new servers
- Installing and configuring software
- Setting up databases and storage
- Configuring monitoring and backups
- Testing the deployment

With multi-tenant services, onboarding a new customer is as simple as updating a configuration table or issuing API credentials.

### Resource Sharing Benefits

Resource sharing is a central benefit in multi-tenant systems. A **multi-tenant system** handles multiple workloads, such as work from multiple customers at once. This system can also handle low priority, non-urgent workloads along with high-priority, urgent workloads.

A **single-tenant system**, on the other hand, handles workloads from a single customer. Each customer needs their own dedicated infrastructure.

Resource sharing is a powerful concept behind SoA and cloud architecture, leading to multiple benefits:

### 1. Infrastructure Cost Savings

Multi-tenant systems achieve dramatically higher resource utilization than single-tenant systems. Here's why:

Single-tenant systems must be provisioned for peak load plus headroom (often 2x peak). If a customer's peak load is 1000 requests/second and average load is 100 requests/second, a single-tenant system wastes capacity 90% of the time.

Multi-tenant systems leverage the statistical reality that **different customers' peak loads rarely coincide**. If you have 100 customers each with 10% utilization on average:
- Single-tenant: Need 100 fully provisioned systems
- Multi-tenant: Need approximately 10-15 systems (depending on peak variance)

That's an 85-90% reduction in required infrastructure.

| Architecture | Customers | Avg Utilization | Peak Provision | Total Servers Needed |
|--------------|-----------|----------------|----------------|---------------------|
| Single-Tenant | 100 | 10% | 2x peak | 100 |
| Multi-Tenant | 100 | 10% | Aggregate + headroom | 15-20 |
| Savings | - | - | - | 80-85% fewer servers |

### 2. Operational Cost Reduction

Lower operational costs follow from managing one fleet instead of many. Instead of operating hundreds of separate database clusters, teams operate unified fleets with:
- Automated scaling
- Centralized monitoring
- Batch upgrades and patches
- Shared maintenance windows

This operational leverage means **small teams can support massive scale**. A team of 10 engineers can operate a multi-tenant service supporting thousands of customers, whereas single-tenant systems might require dedicated ops teams per customer.

### 3. Environmental Benefits

Environmental benefits are increasingly important. Higher utilization means fewer servers, which translates directly to:
- Less energy for compute
- Less energy for cooling
- Smaller data center footprint
- Reduced e-waste from retired servers

As cloud providers operate at massive scale, these efficiency gains have meaningful environmental impact. A 2020 study found that moving to cloud infrastructure can reduce carbon emissions by 88% compared to on-premises data centers, largely due to higher utilization from multi-tenancy.

### 4. Degrees of Multi-Tenancy

Just as with databases, there are degrees of multitenancy for general-purpose compute servers:

**Compute Services Sharing Granularity:**

| Service | Sharing Level | Isolation Mechanism |
|---------|--------------|-------------------|
| EC2 | Instance (dedicated) | Hardware virtualization |
| ECS/Fargate | Container | Container isolation |
| Lambda | Sub-second | Firecracker micro-VMs |
| API Gateway | Request | API-level throttling |

With **AWS Lambda**, compute resource sharing happens at a sub-second interval, using **Firecracker lightweight virtualization** for resource isolation. A single physical server might run execution environments for hundreds of different customers within the same minute, switching between them in milliseconds.

With **Amazon API Gateway**, resource sharing is at the API request level. A single API Gateway server might handle requests from thousands of different customers concurrently, relying on sophisticated throttling and admission control to ensure fairness.

Customers of these services benefit from the advantages of multi-tenant systems:
- **Elasticity**: Scale from zero to millions of requests instantly
- **Efficiency**: Pay only for what you use, not for idle capacity
- **Ease of use**: No capacity planning or server management

### The Challenge: Maintaining Fairness

Under the hood, these services work to address the central challenge that comes with multitenancy: **fairness**. How do you ensure that one customer's spike doesn't impact another customer's performance?

## Fairness in Multi-Tenant Systems: Core Concepts

Any multitenancy service works in concert with systems to ensure fairness. **Fairness** means that every client in a multi-tenant system is provided with a single-tenant experience—predictable performance, guaranteed capacity, and isolation from other tenants' behavior.

The systems that ensure fairness in multi-tenant systems are similar to systems that perform **bin-packing algorithms**, which are classic algorithms in computer science. These fairness systems do the following things:

### 1. Placement Algorithms

Perform placement algorithms to find a spot in the fleet for new workload. This is similar to finding a bin with room for the workload.

For example, when a new DynamoDB table is created, the placement system must decide:
- Which physical servers should host partitions for this table?
- How many partitions are needed based on provisioned capacity?
- Which Availability Zones should host replicas?
- How to balance load across the existing fleet?

### 2. Continuous Monitoring and Rebalancing

Continuously monitor the utilization of each workload and each server to move workloads around. This is similar to moving workloads between bins to ensure that no bin is too full.

Amazon's fairness systems track:
- CPU utilization per server and per tenant
- Memory consumption and allocation
- Network I/O patterns
- Request latency distributions
- Throttling rates and quota consumption

When a server becomes overloaded or a tenant outgrows its current placement, the system automatically migrates workloads to maintain balance.

### 3. Fleet-Level Capacity Management

Monitor the overall fleet utilization and add or remove capacity as needed. This is similar to adding more bins when they're all getting full, and removing bins when they're empty.

Auto-scaling at the fleet level ensures that:
- New capacity is added before existing capacity is exhausted
- Scaling happens gradually to avoid thundering herds
- Excess capacity is removed to reduce costs
- Scaling respects budget constraints and quotas

### 4. Elastic Resource Borrowing

Allow workloads to stretch beyond hard-allocated performance boundaries as long as the underlying system isn't being fully utilized, and hold workloads to their boundaries when the system is fully utilized.

This is similar to allowing workloads to stretch within each bin as long as they're not crowding out other workloads.

</Callout>

### Advanced Fairness: Intelligent Co-Location

Advanced fairness systems combine these techniques in interesting ways. For example, a fairness system can:

1. **Monitor utilization patterns** of each workload over time
2. **Estimate compatibility** between workloads (are they likely to spike at the same time?)
3. **Co-locate complementary workloads** that use resources at different times
4. **Enable resource borrowing** when neighbors are idle

As long as one workload isn't fully utilizing its provisioned resources, another workload in the same bin can borrow those resources.

**Example: Complementary Traffic Patterns**

Consider two DynamoDB tables on the same physical server:
- **Table A**: E-commerce application, busy during daytime (9 AM - 9 PM)
- **Table B**: Data processing job, busy at night (10 PM - 6 AM)

These tables have complementary patterns. Table A can borrow Table B's unused capacity during the day, and Table B can borrow Table A's unused capacity at night. Both get better performance than if they were strictly isolated.

### The Challenge: Instantaneous Resource Return

For this resource sharing to work, the borrowing needs to go unnoticed by the workloads. If a workload needs to use all of its provisioned resources, the time to return those borrowed resources needs to be nearly instantaneous.

In addition, workloads need to be moved quickly between bins. If a busy workload grows accustomed to exceeding its provisioned resources by borrowing from a neighbor, but its neighbor changes its behavior and starts using more of its provisioned resources, the busy workload needs to be moved to another bin.

This requires:
- **Fast detection** of resource contention (millisecond-level monitoring)
- **Quick throttling** to return borrowed resources
- **Live migration** capabilities to move workloads without downtime
- **Predictive analytics** to anticipate when migration will be needed

## Load Shedding Plus Fairness

When load increases faster than auto-scaling can add capacity, services face a choice: become slow for all requests, or shed excess load while maintaining consistent performance for accepted requests. Amazon strongly prefers the second option—maintaining predictable performance even during overload.

**Load shedding** means cheaply rejecting work instead of spending limited resources on it. For HTTP services, this typically means returning an immediate 503 error. This fail-fast strategy allows the server to continue making progress on the requests it does accept, buying time for auto-scaling to add capacity.

> "Load shedding is the act of cheaply rejecting work instead of spending limited resources on it. Because returning a load shed response is significantly cheaper than fully processing the request, this approach lets the server continue to offer predictable performance for the requests it decides to fully process."

However, in multi-tenant systems, simple load shedding isn't sufficient for fairness. Load from multiple tenants is typically uncorrelated—each customer has their own usage patterns. If overall load spikes, it's usually driven by a single tenant. Basic load shedding would fail some requests across all tenants, which isn't fair to customers operating within their normal patterns.

**Rate limiting with per-tenant quotas** solves this fairness problem. When traffic increases beyond capacity, only the workload exceeding its quota is rejected. Other tenants continue operating with predictable performance. This approach provides each tenant with a single-tenant experience despite running on shared infrastructure.

The HTTP status code distinction is meaningful. 503 errors (5xx range) indicate server-side problems—the service is overloaded. 429 errors (4xx range) indicate client-side issues—the client is sending more traffic than planned. This distinction helps clients understand whether they should back off temporarily (503) or reduce their overall rate (429).

Note that some AWS services use 503 for rate limiting because they predate the 2012 RFC 6585 that officially added 429. AWS maintains backward compatibility rather than changing behavior that existing clients depend on.

## Quota Visibility, Flexibility, and Bursting

For quota systems to scale operationally, customers need visibility into their quotas and current utilization. When a client exceeds its quota, it returns errors to its own clients, so services provide metrics showing quota values and consumption over time. DynamoDB, for example, publishes CloudWatch metrics showing provisioned throughput and actual consumption.

**Variable-cost APIs** complicate quota enforcement. Some APIs are far more expensive than others. A query returning one 1KB row is much cheaper than one returning 1MB of rows. Services handle this by counting larger responses as multiple requests, sometimes debiting quotas retroactively based on actual cost rather than estimated cost.

This leads to interesting implementation strategies. A service might optimistically count every request as the cheapest possible operation, then after processing, debit the actual cost—potentially pushing the client's quota negative until enough time passes for the bucket to refill.

**Bursting and soft limits** provide flexibility that improves customer experience without sacrificing fairness. Consider a service scaled for 10,000 TPS currently serving 5,000 TPS across all clients. If a client with a 1,000 TPS limit suddenly needs 3,000 TPS, strictly enforcing the limit would reject 2,000 TPS despite available capacity.

Instead, services can allow bursting into unused capacity. If other clients later need their capacity, the bursting client's over-quota requests can be dropped. Critically, this must be observable—the bursting client should know they're exceeding quota and at risk of future errors. Operators should know the service needs scaling or quota adjustment.

Amazon also considers typical customer traffic patterns when designing quotas. EC2 instances are busier at launch than during steady state because operating systems and applications must download and start. By providing generous up-front burst quotas, EC2 reduces boot times while still enabling long-term capacity planning for fairness between workloads.

Some services automatically increase quotas as customers grow. However, quotas used for cost control are intentionally fixed—customers depend on them not changing unexpectedly.

## Implementing Admission Control: Local and Distributed

Admission control systems shape traffic, shed load, and enforce rate-based quotas. Amazon typically implements multiple layers of admission control, using API Gateway, Application Load Balancer, or CloudFront with AWS WAF to absorb surges, with deeper protection from AWS Shield for DDoS.

**Local admission control** is the simplest approach, using the **token bucket algorithm**. A bucket holds tokens, and each admitted request consumes a token. If no tokens are available, requests are rejected. Tokens are added to the bucket at a configured rate up to a maximum burst capacity.

This burst capacity is a double-edged sword. It accommodates natural traffic variability, but if too large, it defeats rate limiting protections. A solution is composing multiple token buckets: one with low rate and high burst, another with high rate and low burst. Checking both allows high burst with bounded burst rate.

<Callout type="tip">
Off-the-shelf implementations like Google Guava's RateLimiter class provide tested local rate limiting. Use proven libraries rather than implementing algorithms from scratch.
</Callout>

**Distributed admission control** is necessary when quotas must be enforced across a horizontally scaled fleet. Amazon uses several approaches:

**Dividing quotas by server count** assumes uniform traffic distribution. If a client has 1,000 TPS quota and there are 10 servers, each server enforces 100 TPS locally. This works well when load balancers distribute requests round-robin but breaks down with connection-based balancing or multiple load balancers where traffic may concentrate on server subsets.

**Consistent hashing to rate tracker servers** uses a separate fleet (like ElastiCache for Redis) where consistent hashing routes throttle keys to specific servers for centralized counting. This handles high-cardinality keys well but creates hot spots when particular keys are requested at very high rates. Solutions require gradually shifting to local admission control for high-throughput keys.

**Asynchronous information sharing** involves servers periodically sharing observed rates for each throttle key across the fleet. This handles non-uniform traffic but involves complex tradeoffs between scalability, accuracy, and operational simplicity.

## High-Cardinality Dimensions and Reactive Control

Some quota dimensions have practically unbounded cardinality—every IPv6 address, every DynamoDB row, every S3 object. No amount of memory can track rates and quotas for unlimited entities.

For these cases, Amazon uses algorithms like **Heavy Hitters**, **Top Talkers**, and **Counting Bloom filters**. These provide accuracy guarantees and error bounds while limiting memory usage. They identify the highest-traffic keys without tracking every possible key.

**CloudWatch Contributor Insights** helps operators analyze high-cardinality traffic patterns, identifying which keys are consuming the most quota and detecting anomalies.

**Reactive admission control** provides flexibility for unexpected scenarios. A buggy client might send malformed requests, a workload might be more expensive than anticipated, or a client might request help filtering accidental traffic. Services need the ability to quickly add rate limit rules based on HTTP headers, URIs, source IP addresses, or other request attributes.

Critically, rule changes must deploy quickly and safely. Configuration loaded only at process startup is too slow for incident response. Dynamic configuration systems must include audit trails and ideally support evaluation mode—verifying rules affect the right traffic before making them live.

## Architectural Patterns to Avoid Quotas

While admission control protects services, Amazon also focuses on helping customers avoid hitting quotas through thoughtful API design. Approaches differ between **control plane** APIs (low-volume management operations like CreateBucket or DescribeInstances) and **data plane** APIs (high-volume operations like GetObject or GetItem).

For data planes, key strategies include:

**Bursting into slack capacity** when the fleet has unused resources
**Auto-scaling quotas** as customers grow in normal usage
**Proactive monitoring and alarming** when customers approach limits

For control planes, where operations don't scale with data volume, Amazon uses architectural patterns to avoid quota-hitting access patterns:

**Change streams** eliminate polling. Instead of repeatedly calling DescribeInstances to detect new launches, customers subscribe to CloudTrail change logs and react to events. This reduces DescribeInstances calls from continuous polling to zero.

**Data export to high-throughput stores** helps customers access large datasets without hitting control plane limits. S3 Inventory exports bucket object lists to JSON manifest files in S3, allowing customers to query millions of objects at data plane throughput instead of paginating through ListObjects at control plane rates.

**Bulk asynchronous APIs** simplify large-scale operations. AWS IoT's Bulk Provisioning APIs let customers upload files with thousands of changes. The service processes them asynchronously and provides result files with successes and failures. Customers avoid writing complex retry logic and don't risk crowding out other control plane operations.

**Metadata projection** eliminates control plane calls entirely. EC2 Instance Metadata Service provides each instance with its own metadata through a local service. Applications avoid remote DescribeInstances calls, which wouldn't scale to millions of instances all querying simultaneously.

<KeyTakeaways>
- **Multi-tenancy enables cloud economics**: Resource sharing provides higher utilization, lower costs, and better environmental efficiency than single-tenant systems, but requires sophisticated fairness mechanisms.
- **Per-tenant quotas protect all tenants**: Rate limiting at a per-tenant granularity ensures that one customer's spike doesn't degrade performance for others, providing a single-tenant experience on shared infrastructure.
- **Bursting improves experience without sacrificing fairness**: Allowing temporary bursts into unused capacity improves customer experience while maintaining protections—as long as burst usage is visible and controllable.
- **Quota visibility is essential**: Customers need metrics showing quota values and consumption to avoid surprises, and services need visibility into quota violations to detect capacity issues.
- **API design can eliminate quota problems**: Change streams, data exports, bulk APIs, and metadata projection help customers accomplish their goals without hitting control plane rate limits.
</KeyTakeaways>
